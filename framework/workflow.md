# AI Harness 워크플로우

> 클로드를 쓰든, 코덱스를 쓰든, 제미나이를 쓰든 —
> 하네스 엔지니어링 기법을 잘 활용하면 어떤 AI 모델을 써도 같은 아웃풋이 나오도록 방향을 강제할 수 있다.

---

## 핵심 개념: 멱등성 (Idempotency)

멱등성이란 입력이 달라져도 결과가 동일하게 수렴하는 성질이다. 하네스 엔지니어링은 이 성질을 AI 아웃풋에 적용한다. 어떤 모델을 쓰더라도, 어떤 프롬프트 표현을 쓰더라도 설계 원칙과 린터 규칙이 최종 결과물을 하나의 방향으로 수렴시킨다.

---

## 전체 플로우 (6단계)

```
1. 요구사항 입력 (비정형 OK)
        ↓
2. 플랜 수립 (미팅 로그 → CPS → PRD)
        ↓
3. 설계 분석 (원칙 정의 → 개념 정의 → 아키텍처)
        ↓
4. 코드 레벨 설계 (도메인 모델 → AI 구현 요청)
        ↓
5. 린터 적용 (커밋 시 자동 강제)
        ↓
6. 이밸루에이션 (조직 맞춤형 품질 모니터링)
```

---

## 1단계 — 요구사항 입력

고객은 비정형적으로 요구사항을 전달해도 된다. FDE(Forward Deployed Engineer)가 그 내용을 듣고 시스템 구성 요소를 구조화한다.

**예시 (배달 플랫폼)**

> 고객: "배달 앱 만들어 주세요"

FDE 분석 결과:
- 고객용 주문 앱
- 배달 기사용 픽업·이동 앱
- 음식점용 주문 수락·조리 완료 앱
- 전체 관리용 어드민

고객은 처음에 "주문 앱 하나면 되지"라고 생각하지만, 대화가 진행되며 나머지 구성 요소의 필요성을 스스로 인식하게 된다. FDE의 역할은 이 탐색을 이끌고 구성 요소를 정의하는 것이다.

---

## 2단계 — 플랜 수립

플랜은 세 가지 구성 요소로 이루어진다.

> 💡 미팅 자체에 AI 에이전트가 실시간으로 참여하는 구조에 대한 아이디어 스케치는 [meeting_agent.md](./meeting_agent.md)를 참고한다.

### 미팅 로그

- 주 2~3회 미팅 (오전·오후 각 1회 = 주 최대 4회)
- 각 미팅마다: 직전 완성 결과물 리뷰 + 금일 목표 확인
- 발산적으로 진행된 대화를 수렴·구조화하여 기록

**미팅 원칙**
1. 고객의 말을 제한하지 않는다 — 모든 맥락이 데이터가 된다
2. FDE 2인이 함께 참여해 서로 제한하려는 습관을 방지한다

### CPS (Context · Problem · Solution)

가장 하이레벨한 기획 프레임워크. 팔란티어 FDE 방법론에서 착안했다.

| 구성 요소 | 내용 |
|-----------|------|
| **Context** | 모두가 공유하는 배경 맥락 |
| **Problem** | 고객이 겪고 있는 문제 |
| **Solution** | 함께 만들어 갈 해결 방향 |

- 미팅 한 번마다 미팅 로그가 생성되고, 누적된 미팅 로그가 하나의 CPS로 수렴된다
- CPS는 메인 브랜치에 계속 쌓이며 업데이트된다
- 고객사에 GitHub 접근 권한을 부여해 실시간으로 공유
- 정식 전달은 POC 기간 중 2회: ①1차 스펙 확정 시 ②프로젝트 완료 후 역기획 문서로

### PRD (Product Requirements Document)

CPS 이후 전통적인 기획 문서로 이행한다. 클라이언트가 엄밀히 리뷰하기보다는 산출물을 보고 하이레벨 피드백을 주는 방식으로 활용된다.

---

## 3단계 — 설계 분석

### 원칙 정의 (Principles)

프로젝트의 대전제를 먼저 세운다. 예시:
- 할루시네이션 없음
- 감사 로그 필수
- 휴먼 인더 루프(Human-in-the-Loop) 적용

### 개념 정의 (Definitions)

고객사와 함께 도메인 용어와 개념을 정의한다. 개념 간 계층구조(hierarchy)가 결정되면 데이터 구조와 아키텍처가 자연스럽게 도출된다.

### 아키텍처 설계

- 도메인 드리븐 디자인(DDD) 채택
- 도메인 모델 식별 → 계층 구조 확정 → 데이터 인터페이스 설계
- AI가 이해할 수 있는 코드 레벨 설계 명세로 전환

> 유저 멘탈 모델도 설계 문서에 포함한다. 사용자가 시스템을 어떻게 인식하는지가 아키텍처 결정에 영향을 준다.

---

## 4단계 — 코드 구현

설계 문서가 완성되면 AI에 구현을 요청한다. 이 단계에서 드는 노력은 기존 대비 매우 줄어든다. 핵심은 어떤 AI 모델을 사용하든 같은 구조로 코드가 나오도록 다음 단계의 린터가 보장한다는 점이다.

---

## 5단계 — 린터 (Linter) 강제화

하네스 엔지니어링의 핵심 도구. 프롬프트 엔지니어링만으로는 항상 동일한 아웃풋을 강제할 수 없기 때문에, **커밋 시점에 린터를 자동 실행**해 규칙 위반을 차단한다.

### 적용 규칙 예시

| 규칙 | 내용 |
|------|------|
| 파일 명명 | 목록 페이지는 도메인 복수형으로 강제 (`restaurants/`, `orders/`) |
| 금지 접미사 | 파일명에 `List`, `Detail` 사용 불가 |
| 임포트 순서 | 알파벳 순 정렬 강제 |
| 임포트 제한 | 허용/불허 파일 명시적 지정 |
| 배럴 파일 | 익스포트 규칙 통일 |

### 린터가 멱등성을 만드는 원리

```
소문자 요구사항 (a~f)  ─┐
                          ├─▶ 린터 통과 ─▶ 동일한 아웃풋
대문자 요구사항 (A~C)  ─┘
```

어떤 표현의 요구사항을 입력하든 린터를 통과한 결과물은 동일한 구조로 수렴된다.

### 장점과 단점

**장점**
- AI 모델 교체 시에도 결과물 일관성 유지
- 유지보수 인계 시 새 개발자가 규칙을 즉시 파악 가능
- 버그 발생 시 해당 모듈만 삭제 후 재생성 가능 (인터페이스가 명확히 정의되어 있으므로)
- 인간 협업을 위한 가독성 확보

**단점**
- AI가 자연스럽게 생성하는 스타일과 충돌할 수 있어 토큰 비효율이 발생할 수 있음

> 이 단점을 넘어서는 장점은 **인간과의 협업**이다. 바이브코딩 결과물을 일반 SI팀에 유지보수로 넘길 때, 린터로 강제된 코드가 있어야 인계가 가능하다.

## 6단계 — 이밸루에이션 (Evaluation)

AI 에이전트가 잘 작동하는지 지속적으로 모니터링한다. "에이전트 도그(Agent Dog)" — Datadog처럼 AI 에이전트의 품질을 매일 추적한다.

### 기본 평가 지표 (RAG 기반)

| 지표 | 내용 |
|------|------|
| Context Relevance | 맥락에 맞는 답변을 하고 있는가 |
| Answer Faithfulness | 소스 데이터에 근거한 답변인가 |
| Answer Relevance | 질문과 관련된 답변인가 |
| Context Recall | 필요한 정보가 누락되지 않았는가 |

### 조직 맞춤형 기준 설정

평가 기준은 조직 문화에 따라 달라진다.

- "100번 중 1번 틀려도 무조건 답해라" vs "10번 틀릴 것 같으면 아예 답하지 마라"
- 어떤 조직이 옳고 그름이 아니라, 해당 조직의 리스크 허용도에 맞게 설정한다

이밸루에이션은 프로젝트 종료 후 내부 팀이 자체 유지보수할 수 있도록 세팅해 남기는 핵심 자산이다.

## 문서 산출물 전략

| 시점 | 내용 |
|------|------|
| POC 중반 (1차 스펙 확정) | 기능 명세 + 분석 과정 문서 전달 |
| POC 완료 후 | 역기획 문서 (최종 산출물 기반으로 재작성) |

- 형식: Markdown → LaTeX 템플릿 → PDF 빌드
- 저장소: GitHub (고객사 초대)
- 비공식 업데이트: 미팅마다 자동 갱신
- 공식 전달: 2회

---

## FDE 팀 내 지식 공유 방식

방법론 통일보다 **멘탈 모델 일치**를 우선한다.

- 모든 산출물은 GitHub 팀 레포에 공개
- 모든 로그는 Notion에 기록
- 커뮤니케이션은 Slack
- 고정 요일에 팀 내 설계 리뷰 세션 운영 → 집단 지성으로 방법론 발전

프로젝트마다 KPI가 다르기 때문에 특정 방법론을 강제하는 것보다, "고객의 성공"이라는 제1 원칙 아래 각자의 강점을 살리는 방식을 채택한다.
