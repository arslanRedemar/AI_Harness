# AI Harness

> 클로드를 쓰든, 코덱스를 쓰든, 제미나이를 쓰든 —
> 하네스 엔지니어링 기법을 잘 활용하면 어떤 AI 모델을 써도 같은 아웃풋이 나오도록 방향을 강제할 수 있다.

---

## 개념

**AI Harness**는 AI 에이전트가 자율적으로 작동하면서도 일관된 방향성과 품질을 유지할 수 있도록 설계된 구조적 제어 체계다. 말 그대로 AI에 "마구(harness)"를 씌우는 것 — AI의 능력을 억제하는 것이 아니라, 그 능력이 올바른 방향으로 집중될 수 있도록 안내하는 것이다.

핵심 목표는 **멱등성(Idempotency)** 확보다. 어떤 AI 모델을 사용하든, 어떤 방식으로 요구사항을 표현하든, 최종 결과물이 동일한 품질과 구조로 수렴되도록 만든다. 이는 설계 원칙, CPS 프레임워크, 린터 강제화, 이밸루에이션의 네 층위가 협력하여 달성된다.

---

## AI에 안정장치를 넣는 방법

AI 에이전트는 강력하지만 방향이 없으면 일관성을 잃기 쉽다. AI Harness는 다음 네 가지 층위에서 안정장치를 구현한다.

### 1. 설계 원칙 (Design Principles)

모든 것의 대전제. 코드 한 줄이 작성되기 전에 프로젝트의 불변 원칙을 먼저 정의한다.

- 할루시네이션 허용 여부, 감사 로그 필수 여부, 휴먼 인더 루프 적용 방식 등
- 원칙 → 개념 정의(Definitions) → 계층 구조(Hierarchy) → 아키텍처 → 코드 순서로 하향 전개
- 상위 원칙이 정해지면 하위 결정들은 자연스럽게 수렴된다

### 2. CPS 프레임워크 (Context · Problem · Solution)

비정형적인 고객 요구사항을 구조화하는 사고 프레임워크. 팔란티어 FDE 방법론에서 착안했다.

```
Context  → 모두가 공유하는 배경 맥락
Problem  → 고객이 실제로 겪고 있는 문제
Solution → 함께 만들어 갈 해결 방향
```

미팅 로그가 누적될수록 CPS는 점점 정밀해진다. 불명확한 요구사항은 회의 안건으로 목록화되고, 매 미팅마다 구체화된다.

### 3. 린터 강제화 (Linter Enforcement)

하네스 엔지니어링의 핵심 도구. 프롬프트 엔지니어링만으로는 항상 동일한 아웃풋을 보장할 수 없다. 커밋 시점에 린터를 자동 실행해 코드 구조 규칙을 강제한다.

```
소문자 요구사항 입력  ─┐
                        ├─▶ 린터 통과 ─▶ 동일한 아웃풋
대문자 요구사항 입력  ─┘
```

적용 규칙 예시:
- 목록 페이지 파일명은 도메인 복수형으로만 허용 (`restaurants/`, `orders/`)
- 파일명에 `List`, `Detail` 접미사 사용 금지
- 임포트 알파벳 순 정렬, 허용/불허 임포트 명시, 배럴 파일 규칙 통일

린터 덕분에 어떤 AI 모델을 교체하더라도, 누가 유지보수를 넘겨받더라도 동일한 코드 구조가 유지된다.

### 4. 이밸루에이션 (Evaluation)

만들어진 AI 에이전트가 지속적으로 잘 작동하는지 추적한다. "에이전트 도그(Agent Dog)" — Datadog처럼 AI 품질을 매일 모니터링한다.

범용 RAG 평가 지표(Context Relevance, Faithfulness, Answer Relevance, Recall)를 기반으로, 각 조직의 리스크 허용도에 맞게 기준을 커스터마이징한다. 이밸루에이션 프레임은 프로젝트 완료 후 내부 팀에 이관되는 핵심 자산이다.

## 전체 워크플로우

```
비정형 요구사항 입력
        ↓
FDE가 시스템 구성 요소 정의
        ↓
미팅 로그 → CPS → PRD (점진적 구체화)
        ↓
설계 원칙 → 개념 정의 → 아키텍처
        ↓
도메인 모델 기반 코드 구현
        ↓
린터 자동 강제 (커밋 시)
        ↓
이밸루에이션 (지속 모니터링)
```

자세한 단계별 내용은 [workflow.md](./framework/workflow.md)를 참고한다.
각 단계에서 필요한 클로드 스킬 및 에이전트 목록은 [skills_and_agents.md](./framework/skills_and_agents.md)를 참고한다.

## 왜 이것이 중요한가

AI 코딩 도구가 보편화되면서 초기 구현 자체는 쉬워졌다. 문제는 그 다음이다.

- AI가 생성한 코드를 일반 개발팀에 유지보수로 넘길 수 있는가?
- 6개월 후 다른 사람이 이어받았을 때 동일한 품질을 유지할 수 있는가?
- AI 모델이 교체되어도 프로세스는 유효한가?

하네스 엔지니어링은 이 세 질문에 모두 "예"라고 답할 수 있도록 설계된다. 특정 모델이나 개인의 역량에 의존하는 것이 아니라, 구조 자체가 품질을 보장하게 만드는 것이 목표다.

## 이 리포지토리에 대해

Claude Code 환경에서 동작하는 AI Harness 구조를 설계하고 문서화하기 위한 공간이다. FDE(Forward Deployed Engineer)가 엔터프라이즈 AX 현장에서 검증한 방법론을 기반으로, 반복 가능하고 신뢰할 수 있는 AI 구축 프로세스를 축적한다.
