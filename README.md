# AI Harness

## 개념

**AI Harness**는 AI 에이전트가 자율적으로 작동하면서도 일관된 방향성과 품질을 유지할 수 있도록 설계된 구조적 제어 체계다. 말 그대로 AI에 "마구(harness)"를 씌우는 것—AI의 능력을 억제하는 것이 아니라, 그 능력이 올바른 방향으로 집중될 수 있도록 안내하는 것이다.

자율 주행 차량이 센서와 규칙 체계로 스스로를 통제하듯, AI Harness는 에이전트가 수행하는 각 작업 단계마다 판단 기준, 출력 형식, 검증 절차를 명시적으로 정의하여 예측 가능하고 신뢰할 수 있는 결과를 만들어낸다.

---

## AI에 안정장치를 넣는 방법

AI 에이전트는 강력하지만 방향이 없으면 일관성을 잃기 쉽다. AI Harness는 다음 세 가지 층위에서 안정장치를 구현한다.

### 1. 입력 제어 (Input Framing)

에이전트에게 전달되는 요구사항이 아무리 비정형적이더라도, Harness는 이를 구조화된 형태로 변환한다.

- 목적(Goal), 제약(Constraint), 성공 기준(Success Criteria)을 명시적으로 분리
- 불명확한 부분을 자동으로 감지하여 회의 안건으로 목록화
- 모호한 요청이 에이전트에 그대로 전달되는 것을 차단

### 2. 스킬 기반 실행 (Skill-Based Execution)

에이전트가 수행하는 모든 작업은 목적이 명확히 정의된 개별 스킬 단위로 분리된다. 각 스킬은 독립적으로 검증 가능하며, 스킬 간 의존성을 명시함으로써 출력 오류가 전파되는 것을 방지한다.

```
요구사항 입력
    └─▶ 실현 가능성 분석 스킬
    └─▶ 효과 예측 스킬
    └─▶ 기술 난이도 산출 스킬
    └─▶ 종합 제안서 작성 스킬
```

### 3. 점진적 구체화 (Progressive Concretization)

에이전트가 한 번에 완벽한 결과를 내놓도록 기대하지 않는다. 대신 단계별로 정보를 보강하며 정확도를 높이는 구조를 사용한다.

- **1단계**: 추상적 요구사항 → 초안 분석 + 불확실 항목 식별
- **2단계**: 미팅 로그(Context / Problem / Solution) → 요구사항 보강
- **3단계**: PRD(Product Requirements Document) → 확정된 실행 계획

각 단계의 출력이 다음 단계의 입력이 되며, 에이전트는 이전 컨텍스트를 누적하여 판단의 근거로 삼는다.

## 워크플로우 개요

자세한 내용은 [workflow.md](./workflow.md)를 참고한다.

### 1단계 — 사전 분석

고객사 미팅 이전에 커미션 내용을 기반으로 AI 에이전트가 다음을 수행한다.

| 분석 항목 | 내용 |
|-----------|------|
| 실현 가능성 | 도메인 문서 분석을 통한 기술적 타당성 검토 |
| 기대 효과 | 업무 효율 향상 수치 예측 |
| 기술 난이도 | 구현 복잡도 및 소요 기간 산출 |
| 비용 예측 | API 사용 비용 및 전체 예산 규모 추정 |

### 2단계 — 미팅 보조 및 기록 관리

미팅 로그를 **Context / Problem / Solution** 구조로 기록하고, 이를 PRD 작성으로 연결한다. 불명확했던 요구사항은 이 단계를 거치며 실행 가능한 스펙으로 구체화된다.

## 목적

이 리포지토리는 AI Harness 구조를 설계하고 문서화하기 위한 공간이다. Claude Code 환경에서 동작하는 에이전트 스킬과 워크플로우 지시사항을 점진적으로 축적하며, 반복 가능하고 신뢰할 수 있는 AX(AI Transformation) 컨설팅 프로세스를 구축하는 것을 목표로 한다.
