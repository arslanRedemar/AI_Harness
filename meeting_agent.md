# 미팅 인텔리전스 에이전트 (Meeting Intelligence Agent)

> 아이디어 스케치 — AX 팀과 고객사 간 미팅에 AI 에이전트가 실시간으로 참여하는 구조

---

## 배경과 문제 인식

FDE 방식의 컨설팅은 주 2~3회 미팅을 반복한다. 이 과정에서 다음과 같은 문제가 발생한다.

- **공백 문제**: 미팅 간격이 길어지면 이전에 결정된 내용, 맥락, 고객의 발언을 잊기 쉽다
- **추상적 요구 처리**: 고객은 막연하고 추상적인 요구를 하는 경우가 많다. FDE가 즉각적으로 구체화하거나 레퍼런스를 찾기 어렵다
- **문서화 지연**: 미팅 중 대화에 집중하면 기록이 늦어지고, 기록에 집중하면 대화 흐름을 놓친다
- **컨텍스트 단절**: 새 참여자나 이전 논의를 잊은 참여자가 같은 주제를 반복 논의하게 된다
- **도메인 지식 공백**: FDE가 모든 업종의 전문가일 수 없다. 금융·의료·물류 등 고객사 도메인에 대한 즉각적 이해가 필요하다

---

## 제안: 미팅 전용 에이전트

미팅 전·중·후 세 단계에 걸쳐 작동하는 에이전트. 과거 미팅 기록을 컨텍스트로 보유하고, 현재 대화를 실시간으로 이해하며, 인터넷에 연결되어 추상적 요구에 즉각 반응한다.

---

## 작동 구조

### Phase 1 — 미팅 전: 브리핑

미팅 시작 전 에이전트가 참여자에게 브리핑을 제공한다.

```
[에이전트 브리핑 예시]

📋 지난 미팅 요약 (3월 20일)
  - 결정: 기사 앱과 고객 앱을 별도 레포로 분리하기로 함
  - 보류: 음식점 앱의 실시간 알림 방식 (FCM vs WebSocket) 미결
  - 고객 요청: "주문 취소 정책을 좀 더 유연하게" — 구체화 필요

📌 오늘 미팅 예정 안건
  1. 보류된 실시간 알림 방식 결정
  2. 어드민 권한 체계 초안 리뷰
  3. 고객 요청 사항 구체화

⚠️  주의: 마지막 미팅 이후 12일 경과. 아래 항목을 재확인 권장
  - 어드민 접근 권한 범위 (미확정 상태)
```

**브리핑 구성 요소**

| 항목 | 내용 |
|------|------|
| 지난 미팅 요약 | 결정 사항, 보류 항목, 고객 발언 중 중요한 것 |
| 누적 미결 항목 | 여러 미팅에 걸쳐 해결되지 않은 항목 목록 |
| 오늘 안건 | 예정된 논의 주제 |
| 공백 경보 | 마지막 미팅으로부터 경과 시간이 길 경우 핵심 재확인 항목 제시 |
| 고객 성향 메모 | 이전 미팅에서 파악된 고객의 관심사·민감 주제·의사결정 패턴 |

---

### Phase 2 — 미팅 중: 실시간 동반

미팅이 진행되는 동안 에이전트가 대화를 듣고 세 가지 방식으로 개입한다.

#### 2-1. 실시간 문서화

대화 내용을 실시간으로 전사하고 CPS 구조에 맞게 분류한다.

```
[실시간 기록 예시]

14:03 고객: "앱이 느리면 안 되는데..."
      → [Problem] 응답 속도에 대한 우려 감지
      → [Context] 고객사는 점심 피크타임 트래픽 집중 환경

14:07 FDE: "캐싱 전략을 어떻게 할지..."
      → [Solution 후보] CDN 캐싱 / Redis 캐싱 / 정적 자산 분리
```

#### 2-2. 실시간 서제스천 (Suggestions)

에이전트가 대화 흐름을 분석하고 FDE에게 조용히 제안을 건넨다.

```
[서제스천 예시]

💡 고객이 "빠른 배달"을 3회 이상 언급 → 지연 시간(latency) SLA를 명시적으로 정의할 것을 제안

💡 현재 논의가 이전 미팅(3/20)에서 보류된 FCM vs WebSocket 주제와 연결됨
   → 당시 결정 보류 이유: "비용 비교 후 결정" → 지금 결정해도 좋은 시점

⚠️  고객 발언이 3/15 미팅에서 합의한 범위를 벗어남
   → "앱 내 결제"는 1차 스펙에서 제외하기로 했음. 재확인 필요
```

#### 2-3. 도메인 지식 RAG 조회

고객이 업종 특유의 용어나 규정을 언급할 때, 에이전트가 사전에 구축된 도메인 지식 베이스에서 관련 맥락을 즉각 검색해 FDE에게 제공한다.

```
[RAG 조회 예시 — 금융 도메인]

고객: "망분리 때문에 클라우드 연동이 어려울 것 같아서요"

에이전트 반응:
  📚 도메인 KB 검색: "금융권 망분리 규정"
  → 전자금융감독규정 제15조: 내부망·외부망 분리 의무
  → 금융보안원 클라우드 이용 가이드라인 (2024): 비중요정보는
    외부 클라우드 허용, 단 암호화 및 접근통제 조건 충족 시
  → 유사 사례: 토스, 카카오뱅크 — 하이브리드 클라우드 구조로 우회

  FDE 서제스천:
  "망분리 환경에서는 비중요 데이터 처리 레이어만 퍼블릭 클라우드로
   분리하는 하이브리드 구조가 가능합니다. 어떤 데이터가 내부망에
   있어야 하는지 먼저 분류해보시겠어요?"
```

```
[RAG 조회 예시 — 의료 도메인]

고객: "환자 데이터가 들어가서 HIPAA 대응이 필요한데요"

에이전트 반응:
  📚 도메인 KB 검색: "HIPAA 기술적 보호조치 요건"
  → PHI(Protected Health Information) 암호화 필수
  → 접근 로그 6년 보관 의무
  → BAA(Business Associate Agreement) 체결 필요
  → AWS, GCP, Azure 모두 HIPAA Eligible 서비스 제공

  FDE 서제스천:
  "HIPAA BAA 체결이 된 클라우드 서비스를 써야 합니다.
   데이터 저장 레이어와 처리 레이어를 분리 설계하면
   규정 준수 범위를 최소화할 수 있습니다."
```

#### 2-4. 추상적 요구 즉각 구체화 (Web-connected)

고객이 추상적인 요구를 할 때 에이전트가 인터넷을 검색해 즉각 레퍼런스를 제공한다.

```
[구체화 예시]

고객: "쿠팡이츠처럼 빠릿빠릿하게 해주세요"

에이전트 반응:
  🔍 웹 검색 결과 기반 분석
  - 쿠팡이츠 평균 앱 응답속도: ~200ms (국내 배달앱 벤치마크 기준)
  - 주요 기법: 낙관적 UI 업데이트, 프리페칭, WebSocket 실시간 상태 동기화
  - 제안 질문: "주문 상태 업데이트 주기를 몇 초로 설정할까요?"
  → 추상적 "빠릿빠릿"을 측정 가능한 지표로 전환
```

### Phase 3 — 미팅 후: 자동 문서화

미팅 종료 후 에이전트가 자동으로 산출물을 생성한다.

| 산출물 | 내용 |
|--------|------|
| 미팅 로그 | 전사본 + CPS 분류 결과 |
| 결정 사항 | 이번 미팅에서 확정된 항목 |
| 미결 항목 | 다음 미팅으로 이월되는 보류 사항 |
| 액션 아이템 | 담당자별 할 일 목록 |
| CPS 업데이트 | 누적 CPS 문서에 이번 미팅 내용 반영 |
| 다음 미팅 안건 초안 | 미결 항목 + 예상 다음 주제 |

## 기술 구성 요소

```
┌──────────────────────────────────────────────────────────┐
│                Meeting Intelligence Agent                 │
│                                                           │
│  ┌─────────────┐    ┌──────────────────────┐             │
│  │  Long-term  │    │   Real-time Engine   │             │
│  │   Memory    │◀───│  (STT + 분석 루프)   │             │
│  │ (미팅 로그) │    └──────────────────────┘             │
│  └─────────────┘              │                           │
│         │                     ▼                           │
│  ┌──────────────┐   ┌──────────────────┐                 │
│  │  Domain KB   │   │  Context Merger  │                 │
│  │  (RAG)       │──▶│ (과거+현재+도메인) │                 │
│  │              │   └────────┬─────────┘                 │
│  │ • Global KB  │            │                            │
│  │ • Project KB │   ┌────────┴──────────────────┐        │
│  │ • Session KB │   ▼         ▼          ▼       ▼       │
│  └──────────────┘ [브리핑] [서제스천] [RAG조회] [웹검색]  │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### 도메인 KB (Knowledge Base) 3계층 구조

| 계층 | 내용 | 갱신 주기 |
|------|------|----------|
| **Global KB** | 산업 공통 도메인 지식 (금융·의료·물류·커머스 등 업종별 규정, 표준, 용어) | 분기 |
| **Project KB** | 프로젝트 전용 문서 (고객사 내부 정책, 기술 스펙, 계약 범위, 이전 유사 프로젝트 패턴) | 미팅마다 |
| **Session KB** | 현재 미팅 실시간 누적 (오늘 발언, 결정, 등장한 신규 용어) | 실시간 |

### RAG vs 웹 검색 — 언제 무엇을 쓰나

```
고객 발언 분석
      │
      ├─▶ 업종 규정·내부 정책·계약 범위 관련?
      │         └─▶ Domain KB (RAG) 조회
      │               신뢰도 높음, 사전 검증된 정보
      │
      └─▶ 경쟁사 벤치마크·최신 기술 트렌드·외부 레퍼런스 관련?
                └─▶ 웹 검색
                      실시간 정보, 신뢰도 검증 필요
```

| 구분 | RAG (Domain KB) | 웹 검색 |
|------|----------------|---------|
| 정보 출처 | 사전 구축된 내부 문서 | 공개 인터넷 |
| 신뢰도 | 높음 (검증된 문서) | 검증 필요 |
| 응답 속도 | 빠름 | 보통 |
| 주요 용도 | 규정·정책·도메인 용어·프로젝트 맥락 | 벤치마크·사례·트렌드 |
| 예시 | "망분리 규정이 뭔지" | "쿠팡이츠 응답속도" |

### 필요 도구

| 도구 | 용도 |
|------|------|
| STT (Speech-to-Text) | 실시간 음성 → 텍스트 전환 |
| Vector DB | Domain KB 임베딩 저장 및 유사도 검색 (예: pgvector, Pinecone) |
| Long-term Memory | 과거 미팅 로그 전체를 검색 가능한 형태로 저장 |
| Web Search | 추상적 요구에 대한 실시간 레퍼런스 탐색 |
| CPS Document Store | 누적 CPS 문서 읽기·쓰기 |
| Notification Channel | FDE에게 서제스천 전달 (화면 표시 또는 이어폰) |

## 스킬 및 에이전트 정의

### 스킬

| 스킬명 | 설명 |
|--------|------|
| `pre-meeting-briefer` | 과거 미팅 로그 전체를 읽고 브리핑 문서 생성 |
| `gap-alert-generator` | 마지막 미팅 이후 경과 시간 기반 재확인 항목 추출 |
| `realtime-cps-tagger` | 실시간 발화를 Context / Problem / Solution으로 분류 |
| `suggestion-engine` | 대화 흐름 분석 후 FDE에게 제안 생성 |
| `contradiction-detector` | 현재 발언이 과거 결정 사항과 충돌하는지 감지 |
| `domain-rag-retriever` | 발화에서 도메인 키워드를 감지하고 Domain KB에서 관련 맥락 검색 |
| `kb-ingestion-pipeline` | 문서(PDF·MD·URL)를 청킹·임베딩하여 Domain KB에 적재 |
| `kb-gap-detector` | 미팅 중 등장한 용어 중 KB에 없는 항목을 감지하여 후처리 적재 목록 생성 |
| `abstract-request-resolver` | 추상적 고객 요구를 웹 검색으로 구체화 |
| `post-meeting-documenter` | 미팅 종료 후 결정 사항, 미결 항목, 액션 아이템 자동 생성 |

### 에이전트

| 에이전트명 | 구성 스킬 | 설명 |
|------------|-----------|------|
| `pre-meeting-agent` | `pre-meeting-briefer` + `gap-alert-generator` | 미팅 시작 전 브리핑 패키지 생성 |
| `live-meeting-agent` | `realtime-cps-tagger` + `suggestion-engine` + `contradiction-detector` + `domain-rag-retriever` + `abstract-request-resolver` | 미팅 진행 중 실시간 동반 (RAG 포함) |
| `post-meeting-agent` | `post-meeting-documenter` + `cps-updater` + `kb-gap-detector` | 미팅 종료 후 문서 생성·CPS 갱신·KB 보강 항목 추출 |
| `kb-management-agent` | `kb-ingestion-pipeline` + `kb-gap-detector` | Domain KB 구축 및 지속 보강 |
| `meeting-intelligence-agent` | 위 네 에이전트 통합 | 미팅 전·중·후 전 과정을 하나의 세션으로 관리 |

## 구현 시 고려사항

**개인정보 및 보안**
- 고객사 대화 내용이 외부 LLM API로 전송될 수 있음 → 온프레미스 모델 또는 데이터 처리 계약 필요
- 미팅 녹음에 대한 참여자 동의 절차 필요

**UX 설계**
- 서제스천이 너무 많으면 오히려 방해가 됨 → 중요도 필터링 필수
- FDE가 대화 중 화면을 보기 어려운 환경이면 이어폰 기반 음성 전달 방식 고려

**데이터 누적 전략**
- 미팅 로그가 쌓일수록 에이전트의 컨텍스트 이해도가 높아짐
- 초기 미팅(데이터 없음)과 성숙 단계(데이터 풍부)의 에이전트 행동 차이를 설계에 반영

**Domain KB 구축 전략**
- 프로젝트 시작 전 고객사 도메인에 해당하는 문서를 선제적으로 적재 (규정집, 내부 정책, 기술 스펙)
- 미팅 중 KB에 없는 용어 등장 시 `kb-gap-detector`가 감지 → 미팅 후 자동으로 적재 후보 목록 생성
- Global KB는 팀 공용으로 유지하고, Project KB는 프로젝트마다 독립 네임스페이스로 격리
