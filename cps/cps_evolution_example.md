# CPS 정밀화 과정 예시

> Context / Problem / Solution 구조가 미팅 로그 누적에 따라 어떻게 구체화되는지 보여주는 시나리오.

---

## 시나리오 설정

| 항목 | 내용 |
|------|------|
| 프로젝트 | 배달 플랫폼 앱 개발 (기사 앱 + 고객 앱 + 어드민) |
| 고객사 | 서울 강남구 중심 지역 배달 스타트업 |
| 현황 | 카카오채널로 수동 주문 접수 중 → 디지털 전환 첫 프로젝트 |
| 미팅 주기 | 격주, 총 6주 4회 |

---

## Meeting 1 — 킥오프 (누적 데이터 없음)

에이전트는 이번 발화에서 직접 추출하는 것 외에 아무것도 할 수 없다.

```
[원발화]
고객: "앱이 느려지면 안 돼요. 배달 특성상 실시간이 중요하거든요."
```

```yaml
# CPS — Meeting 1

Context:
  - 배달 앱 프로젝트 (세부 구조 불명확)
  - "실시간이 중요" (SLA 수치 미정의)

Problem:
  - 성능 저하 우려 (막연)

Solution:
  - (없음 — 논의 전)
```

**에이전트 역할**: 발화 받아쓰기. 일반적 배달 앱 아키텍처 참고 수준.

---

## Meeting 2 — 2주 후 (로그 1회 누적)

지난 미팅 로그와 이번 발화가 결합된다. Context가 구체화되기 시작한다.

```
[이전 로그에서 학습된 것]
- 고객사 배달망: 서울 강남구 중심
- 피크타임: 점심 12-13시, 저녁 18-20시 집중
- 현재 주문 수단: 카카오채널 → 이번이 첫 앱 개발

[원발화]
고객: "점심시간에 주문이 몰리는데 그때 절대 안 터져야 해요."
```

```yaml
# CPS — Meeting 2

Context:
  - 앱 구조: 기사 앱 / 고객 앱 분리
  - 피크타임: 12-13시, 18-20시 트래픽 집중
  - 예상 동시 주문량: 미확인 (카카오채널 기준 추정 필요)

Problem:
  - 피크타임 동시 주문 급증 시 서버 다운 리스크
  - SLA 기준 미정의 ("절대 안 터져야"의 수치화 필요)

Solution 후보:
  - Auto Scaling (Cloud-native)
  - 주문 Queue 분리 처리
  → ⏸ 결정 보류: 예상 동시 주문량 확인 후 재논의
```

```
💡 에이전트 서제스천:
  "피크타임 기준 동시 주문량을 수치로 정의해야
   Auto Scaling 임계값과 Queue 전략을 결정할 수 있습니다.
   카카오채널 일일 주문 건수를 공유해주실 수 있을까요?"
```

---

## Meeting 3 — 4주 후 (로그 2회 누적)

Context가 풍부해지면서 Problem이 추상에서 측정 가능한 형태로 바뀐다.

```
[이전 로그에서 학습된 것]
- 예상 동시 주문량: 피크타임 분당 200건 (카카오채널 통계 기준)
- 기사 수: 현재 50명, 6개월 내 150명 목표
- 주문 상태 흐름: 접수 → 기사 배정 → 픽업 → 배달완료 (4단계)
- FCM vs WebSocket 논쟁: 비용 비교 후 결정으로 보류 (Meeting 2)

[원발화]
고객: "기사가 주문 받고 나서 위치 업데이트가 느리다는 불만이 많을 것 같아요."
```

```yaml
# CPS — Meeting 3

Context:
  - 기사 앱: GPS 위치를 고객 앱 + 어드민에 실시간 전송
  - 동시 접속 기사 수: 피크타임 기준 50-150명
  - 기술 결정 미결: FCM vs WebSocket (Meeting 2 보류)
  - 고객 앱 사용 패턴: 배달 상태 화면을 능동적으로 새로고침

Problem:
  - 기사 위치 업데이트 지연 → 고객 불만으로 직결
  - FCM 푸시 지연 5-30초 → 실시간 위치 전달 부적합 가능성
  - 150명 기사 동시 GPS 스트리밍 시 서버 부하 미검증

Solution:
  - WebSocket 채택 (FCM 대비 실시간성 우위)
  - 기사 위치 업데이트: 3초 간격, 이동 중일 때만 전송 (정지 시 스킵)
  - Redis pub/sub로 위치 이벤트 분산 처리
  → ✅ FCM vs WebSocket 논쟁 종결: WebSocket으로 확정
```

```
⚠️ 에이전트 감지:
  "WebSocket 결정은 Meeting 2에서 보류된 안건입니다.
   오늘 확정 시 이후 인프라 비용 산정에 즉시 반영이 필요합니다."
```

## Meeting 4 — 6주 후 (로그 3회 누적)

에이전트가 고객의 발언 패턴, 의사결정 스타일, 기술 제약까지 내재화한다.
Problem을 **고객이 말하기 전에** 먼저 감지한다.

```
[이전 로그에서 학습된 것]
- 고객 의사결정 패턴: 숫자/데이터 기반 설득에 잘 반응
- 고객 민감 주제: 비용 언급 시 즉각 경계 → 절충안 먼저 제시 필요
- 확정된 결정: WebSocket, 기사/고객 앱 레포 분리, Redis 도입
- 미결 항목: 주문 취소 정책, 어드민 권한 체계
- 기술 스택: React Native + Node.js + PostgreSQL + Redis

[원발화]
고객: "어드민에서 기사 위치를 한 화면에 다 볼 수 있어야 해요."
```

```yaml
# CPS — Meeting 4

Context:
  - 어드민 사용자: 운영팀 3-5명, 비기술직
  - 기사 최대 150명 동시 GPS 스트리밍 (WebSocket, 확정)
  - 위치 이벤트: Redis pub/sub로 분산 처리 중 (확정)
  - 어드민 요구: 지도 기반 실시간 기사 위치 표시

Problem:
  - 150개 WebSocket 스트림을 어드민 단일 클라이언트가 동시 수신
    → 브라우저 DOM 업데이트 병목 가능성
  - 지도 라이브러리 미결 (Naver / Kakao / Google)
  - 비기술직 운영팀 대상 → UI 복잡도 최소화 필수

Solution:
  - 어드민은 WebSocket 직접 연결 대신 SSE(Server-Sent Events)로
    서버 집계 위치 스냅샷 수신 (3초 배치)
  - 지도 라이브러리: Kakao Map
    → 국내 좌표 정확도 우위, 기존 카카오채널 연동 고객사에 친숙
  - 기사 아이콘 클러스터링: 줌 레벨에 따라 자동 그룹핑
    → 150명 동시 표시 시 렌더링 부하 최소화
```

```
💡 에이전트 선제 서제스천 (고객 언급 전):
  "Kakao Map은 월 30만 건 무료, 초과 시 건당 과금 구조입니다.
   어드민 조회 빈도를 미리 산정해두는 것을 권장드립니다."

  (← 고객이 비용에 민감하다는 패턴을 3회 미팅에서 학습한 결과)
```

## CPS 정밀화 비교

| 차원 | Meeting 1 | Meeting 4 |
|------|-----------|-----------|
| **Context** | "배달 앱, 실시간 중요" | "피크 분당 200건, 기사 150명, WebSocket 확정, Redis pub/sub, 운영팀 비기술직 3-5명" |
| **Problem** | "성능 저하 우려 (막연)" | "어드민 단일 클라이언트 150개 WebSocket 수신 → DOM 병목" |
| **Solution** | (없음) | "SSE 집계 스냅샷 + Kakao Map 클러스터링 + 비용 사전 산정 권고" |
| **에이전트 역할** | 발화 받아쓰기 | 고객 언급 전 선제 리스크 감지 |

## 무엇이 CPS를 정밀하게 만드는가

```
Meeting N의 CPS 품질
  = 이번 발화
  + 누적된 확정 결정  (무엇이 이미 결론났는가)
  + 누적된 보류 항목  (무엇이 아직 열려있는가)
  + 고객 발언 패턴    (어떤 프레임에 잘 반응하는가)
  + 기술 스택 제약    (어떤 선택지가 이미 닫혀있는가)
  + 프로젝트 수치 기준선  (분당 200건, 150명, 3초 간격...)
```

초기에는 에이전트가 **기록자** 역할을 한다.
로그가 쌓일수록 에이전트는 고객이 말하기 전에 **문제를 먼저 보는 동반자** 역할로 전환된다.

이것이 CPS 누적의 실질적 가치다.

---

## 파생 효과: CPS가 가능하게 하는 것

| 기능 | 의존하는 CPS 데이터 |
|------|-------------------|
| 브리핑 자동 생성 | 누적 확정 결정 + 보류 항목 |
| 모순 감지 | 과거 Solution vs 현재 발화 비교 |
| 선제 서제스천 | 고객 발언 패턴 + 기술 제약 조합 |
| 다음 안건 초안 | 미결 보류 항목 목록 |
| 지식 공백 감지 | Context에 없는 신규 용어 등장 감지 |
