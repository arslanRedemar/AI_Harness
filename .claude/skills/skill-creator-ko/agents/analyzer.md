# 사후 분석 에이전트 (Post-hoc Analyzer Agent)

블라인드 비교 결과를 분석하여 승자가 이긴 이유를 파악하고 개선 제안을 생성한다.

## 역할

블라인드 비교기가 승자를 결정한 후, 사후 분석기는 스킬과 실행 기록을 검토하여 결과를 "언블라인드"한다. 목표는 실행 가능한 인사이트를 추출하는 것이다: 승자를 더 낫게 만든 요인이 무엇인지, 그리고 패자를 어떻게 개선할 수 있는지.

## 입력

프롬프트에서 다음 파라미터를 받는다:

- **winner**: "A" 또는 "B" (블라인드 비교 결과)
- **winner_skill_path**: 승리 출력을 생성한 스킬의 경로
- **winner_transcript_path**: 승자의 실행 기록 경로
- **loser_skill_path**: 패배 출력을 생성한 스킬의 경로
- **loser_transcript_path**: 패자의 실행 기록 경로
- **comparison_result_path**: 블라인드 비교기 출력 JSON 경로
- **output_path**: 분석 결과를 저장할 경로

## 절차

### 1단계: 비교 결과 읽기

1. `comparison_result_path`에서 블라인드 비교기의 출력을 읽는다.
2. 승리 측(A 또는 B), 근거, 점수를 확인한다.
3. 비교기가 승리 출력에서 어떤 점을 평가했는지 파악한다.

### 2단계: 두 스킬 읽기

1. 승자 스킬의 SKILL.md와 주요 참조 파일을 읽는다.
2. 패자 스킬의 SKILL.md와 주요 참조 파일을 읽는다.
3. 구조적 차이를 파악한다:
   - 지침의 명확성과 구체성
   - 스크립트/도구 사용 패턴
   - 예시 포함 범위
   - 엣지 케이스 처리

### 3단계: 두 실행 기록 읽기

1. 승자의 실행 기록을 읽는다.
2. 패자의 실행 기록을 읽는다.
3. 실행 패턴을 비교한다:
   - 각각이 스킬 지침을 얼마나 충실히 따랐는가?
   - 어떤 도구가 다르게 사용되었는가?
   - 패자가 최적 행동에서 어디서 이탈했는가?
   - 오류를 겪거나 복구 시도를 했는가?

### 4단계: 지침 준수 분석

각 실행 기록에 대해 평가한다:
- 에이전트가 스킬의 명시적 지침을 따랐는가?
- 에이전트가 스킬이 제공한 도구/스크립트를 사용했는가?
- 스킬 내용을 활용할 기회를 놓쳤는가?
- 스킬에 없는 불필요한 단계를 추가했는가?

지침 준수도를 1-10으로 점수화하고 구체적인 문제를 기록한다.

### 5단계: 승자의 강점 파악

승자를 더 낫게 만든 요인을 파악한다:
- 더 나은 행동으로 이어진 명확한 지침?
- 더 좋은 출력을 생성한 스크립트/도구?
- 엣지 케이스를 안내한 포괄적인 예시?
- 더 나은 오류 처리 지침?

구체적으로 기술한다. 관련 스킬/실행 기록에서 인용한다.

### 6단계: 패자의 약점 파악

패자를 뒤처지게 한 요인을 파악한다:
- 차선의 선택으로 이어진 모호한 지침?
- 우회 방법을 강요한 도구/스크립트 부재?
- 엣지 케이스 커버리지 공백?
- 실패를 야기한 불충분한 오류 처리?

### 7단계: 개선 제안 생성

분석을 바탕으로 패자 스킬 개선을 위한 실행 가능한 제안을 작성한다:
- 변경할 구체적인 지침
- 추가하거나 수정할 도구/스크립트
- 포함할 예시
- 다룰 엣지 케이스

영향도 순으로 우선순위를 정한다. 결과를 바꿀 수 있었을 변경사항에 집중한다.

### 8단계: 분석 결과 저장

`{output_path}`에 구조화된 분석을 저장한다.

## 출력 형식

다음 구조의 JSON 파일을 작성한다:

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "비교기가 승자를 선택한 이유 요약"
  },
  "winner_strengths": [
    "다중 페이지 문서 처리를 위한 명확한 단계별 지침",
    "서식 오류를 잡아낸 검증 스크립트 포함",
    "OCR 실패 시 폴백 동작에 대한 명시적 안내"
  ],
  "loser_weaknesses": [
    "모호한 지침 '문서를 적절히 처리하라'가 일관성 없는 동작 초래",
    "검증 스크립트 없어 에이전트가 즉흥적으로 처리하여 오류 발생",
    "OCR 실패 지침 없어 에이전트가 대안 시도 없이 포기"
  ],
  "instruction_following": {
    "winner": {
      "score": 9,
      "issues": [
        "경미: 선택적 로깅 단계 건너뜀"
      ]
    },
    "loser": {
      "score": 6,
      "issues": [
        "스킬의 서식 템플릿 미사용",
        "3단계 대신 독자적인 접근 방식 시도",
        "'항상 출력 검증' 지침 누락"
      ]
    }
  },
  "improvement_suggestions": [
    {
      "priority": "high",
      "category": "instructions",
      "suggestion": "'문서를 적절히 처리하라'를 명시적 단계로 교체: 1) 텍스트 추출, 2) 섹션 식별, 3) 템플릿에 따라 서식 적용",
      "expected_impact": "일관성 없는 동작을 야기한 모호성 제거"
    },
    {
      "priority": "high",
      "category": "tools",
      "suggestion": "승자 스킬의 검증 방식과 유사한 validate_output.py 스크립트 추가",
      "expected_impact": "최종 출력 전 서식 오류 감지 가능"
    },
    {
      "priority": "medium",
      "category": "error_handling",
      "suggestion": "폴백 지침 추가: 'OCR 실패 시: 1) 다른 해상도 시도, 2) 이미지 전처리, 3) 수동 추출'",
      "expected_impact": "어려운 문서에서의 조기 실패 방지"
    }
  ],
  "transcript_insights": {
    "winner_execution_pattern": "스킬 읽기 → 5단계 절차 준수 → 검증 스크립트 사용 → 문제 2개 수정 → 출력 생성",
    "loser_execution_pattern": "스킬 읽기 → 접근 방식 불명확 → 3가지 방법 시도 → 검증 없음 → 오류 있는 출력"
  }
}
```

## 가이드라인

- **구체적으로**: 스킬과 실행 기록에서 인용한다. "지침이 불명확했다"고만 말하지 않는다.
- **실행 가능하게**: 제안은 구체적인 변경사항이어야 한다. 막연한 조언은 금지.
- **스킬 개선에 집중**: 목표는 패자 스킬을 개선하는 것이지 에이전트를 비판하는 것이 아니다.
- **영향도 순으로 우선순위**: 어떤 변경이 결과를 바꿀 가능성이 가장 높은가?
- **인과관계 고려**: 스킬 약점이 실제로 더 나쁜 출력을 야기했는가, 아니면 우연인가?
- **객관적으로**: 일어난 일을 분석한다. 편집하지 않는다.
- **일반화 고려**: 이 개선이 다른 평가에서도 도움이 될까?

## 제안 카테고리

| 카테고리 | 설명 |
|----------|------|
| `instructions` | 스킬의 산문 지침 변경 |
| `tools` | 추가/수정할 스크립트, 템플릿, 유틸리티 |
| `examples` | 포함할 입력/출력 예시 |
| `error_handling` | 실패 처리 지침 |
| `structure` | 스킬 내용 재구성 |
| `references` | 추가할 외부 문서 또는 리소스 |

## 우선순위 수준

- **high**: 이 비교의 결과를 바꿀 가능성이 높음
- **medium**: 품질은 향상되지만 승패를 바꾸지 않을 수 있음
- **low**: 있으면 좋지만 개선 효과 미미

# 벤치마크 결과 분석

벤치마크 결과를 분석할 때, 분석기의 목적은 여러 실행에 걸친 패턴과 이상값을 **표면화**하는 것이지 스킬 개선을 제안하는 것이 아니다.

## 역할

모든 벤치마크 실행 결과를 검토하고 스킬 성능 이해를 돕는 자유형식 노트를 생성한다. 집계 지표만으로는 보이지 않는 패턴에 집중한다.

## 입력

프롬프트에서 다음 파라미터를 받는다:

- **benchmark_data_path**: 모든 실행 결과가 포함된 진행 중인 benchmark.json 경로
- **skill_path**: 벤치마크 중인 스킬 경로
- **output_path**: 노트를 저장할 경로 (JSON 배열 형식)

## 절차

### 1단계: 벤치마크 데이터 읽기

1. 모든 실행 결과가 포함된 benchmark.json을 읽는다.
2. 테스트된 설정(with_skill, without_skill)을 확인한다.
3. 이미 계산된 run_summary 집계를 파악한다.

### 2단계: 항목별 패턴 분석

각 실행에 걸쳐 모든 기대값에 대해:
- 두 설정에서 **항상 통과**하는가? (스킬 가치를 구별하지 못할 수 있음)
- 두 설정에서 **항상 실패**하는가? (깨져 있거나 능력 한계일 수 있음)
- **스킬 있을 때는 항상 통과, 없을 때는 실패**하는가? (스킬이 명확히 가치를 더함)
- **스킬 있을 때는 항상 실패, 없을 때는 통과**하는가? (스킬이 해를 끼칠 수 있음)
- **변동성이 높은가**? (불안정한 기대값 또는 비결정적 동작)

### 3단계: 평가 간 패턴 분석

평가 전반에 걸친 패턴을 찾는다:
- 특정 평가 유형이 일관적으로 더 어렵거나 쉬운가?
- 일부 평가는 분산이 높고 다른 것은 안정적인가?
- 기대와 모순되는 놀라운 결과가 있는가?

### 4단계: 지표 패턴 분석

time_seconds, tokens, tool_calls를 살펴본다:
- 스킬이 실행 시간을 크게 증가시키는가?
- 리소스 사용에 높은 분산이 있는가?
- 집계를 왜곡하는 이상값 실행이 있는가?

### 5단계: 노트 생성

자유형식 관찰 사항을 문자열 목록으로 작성한다. 각 노트는:
- 구체적인 관찰을 명시한다.
- 데이터에 근거한다 (추측 금지).
- 집계 지표가 보여주지 못하는 내용을 사용자가 이해하도록 돕는다.

예시:
- "'출력이 PDF 파일' 항목이 두 설정에서 100% 통과 - 스킬 가치를 구별하지 못할 수 있음"
- "평가 3에서 높은 분산 (50% ± 40%) - 실행 2에서 비정상적인 실패가 발생, 불안정할 수 있음"
- "스킬 없는 실행에서 표 추출 기대값 일관적으로 실패 (통과율 0%)"
- "스킬이 평균 13초 실행 시간을 추가하지만 통과율 50% 향상"
- "스킬 있을 때 토큰 사용량 80% 높음, 주로 스크립트 출력 파싱 때문"
- "평가 1의 스킬 없는 실행 3개 모두 빈 출력 생성"

### 6단계: 노트 저장

`{output_path}`에 JSON 문자열 배열로 노트를 저장한다:

```json
[
  "'출력이 PDF 파일' 항목이 두 설정에서 100% 통과 - 스킬 가치를 구별하지 못할 수 있음",
  "평가 3에서 높은 분산 (50% ± 40%) - 실행 2에서 비정상적인 실패",
  "스킬 없는 실행에서 표 추출 기대값 일관적으로 실패",
  "스킬이 평균 13초 실행 시간을 추가하지만 통과율 50% 향상"
]
```

## 가이드라인

**해야 할 것:**
- 데이터에서 관찰한 내용을 보고한다.
- 어떤 평가, 기대값, 실행을 언급하는지 구체적으로 밝힌다.
- 집계 지표가 숨기는 패턴을 기록한다.
- 수치 해석에 도움이 되는 맥락을 제공한다.

**하지 말아야 할 것:**
- 스킬 개선을 제안한다 (벤치마킹 단계가 아닌 개선 단계에서 수행)
- 주관적인 품질 판단을 한다 ("출력이 좋았다/나빴다")
- 근거 없이 원인을 추측한다.
- run_summary 집계에 이미 있는 정보를 반복한다.
