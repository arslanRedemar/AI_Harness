# Skill Creator (스킬 제작기)

새로운 스킬을 만들고 반복적으로 개선하기 위한 스킬.

스킬 생성의 전체 과정은 다음과 같다:

- 스킬이 무엇을 해야 하는지, 대략 어떻게 해야 하는지 결정한다.
- 스킬 초안을 작성한다.
- 몇 가지 테스트 프롬프트를 작성하고 해당 스킬에 접근할 수 있는 claude로 실행한다.
- 사용자와 함께 결과를 정성적·정량적으로 평가한다.
  - 실행이 진행되는 동안, 정량적 평가가 없으면 초안을 작성한다 (있으면 그대로 사용하거나 변경이 필요하다고 느끼면 수정). 그런 다음 사용자에게 설명한다 (이미 존재했다면 기존 평가를 설명).
  - `eval-viewer/generate_review.py` 스크립트를 사용하여 사용자가 결과를 검토하고 정량적 지표도 볼 수 있게 한다.
- 사용자의 평가 피드백을 바탕으로 스킬을 재작성한다 (정량적 벤치마크에서 명백한 결함이 보이면 그것도 반영).
- 만족할 때까지 반복한다.
- 테스트 세트를 확장하고 더 큰 규모로 다시 시도한다.

이 스킬을 사용할 때의 역할은 사용자가 이 과정의 어느 단계에 있는지 파악하고 단계를 진행하도록 돕는 것이다. 예를 들어 "X를 위한 스킬을 만들고 싶어"라고 하면, 의미를 좁히고, 초안을 작성하고, 테스트 케이스를 작성하고, 평가 방법을 결정하고, 모든 프롬프트를 실행하고, 반복한다.

반면 이미 스킬 초안이 있다면, 바로 평가/반복 단계로 넘어갈 수 있다.

물론 유연하게 대응해야 하며, 사용자가 "많은 평가가 필요 없어, 그냥 같이 해봐"라고 하면 그렇게 한다.

스킬이 완성된 후 (순서는 유연하게), 스킬 설명 개선기를 실행하여 스킬 트리거를 최적화할 수 있다. 이를 위한 별도의 스크립트가 있다.

## 사용자와의 소통

skill creator는 코딩 용어에 대한 친숙도가 다양한 사람들이 사용할 수 있다. Claude의 능력에 영감을 받은 배관공이 터미널을 열고, 부모님과 조부모님이 "npm 설치 방법"을 구글링하는 시대다. 반면 대부분의 사용자는 컴퓨터에 꽤 익숙할 것이다.

따라서 맥락 단서를 잘 파악하여 소통 방식을 결정한다. 기본 케이스의 예시:

- "evaluation"과 "benchmark"는 경계선에 있지만 OK
- "JSON"과 "assertion"은 사용자가 그것들을 안다는 진지한 단서를 보기 전에는 설명 없이 사용하지 않는다.

의심스러울 때는 용어를 짧게 설명해도 괜찮다.

## 스킬 생성

### 의도 파악

사용자의 의도를 이해하는 것부터 시작한다. 현재 대화에 사용자가 캡처하고 싶은 워크플로우가 이미 포함되어 있을 수 있다 (예: "이것을 스킬로 만들어줘"). 있다면 먼저 대화 기록에서 답을 추출한다 — 사용된 도구, 단계의 순서, 사용자가 한 수정, 관찰된 입출력 형식. 사용자가 공백을 채워야 할 수 있으며, 다음 단계로 넘어가기 전에 확인을 받아야 한다.

1. 이 스킬이 Claude에게 무엇을 할 수 있게 해야 하는가?
2. 언제 이 스킬이 트리거되어야 하는가? (어떤 사용자 문구/맥락)
3. 예상 출력 형식은 무엇인가?
4. 스킬이 작동하는지 확인하기 위한 테스트 케이스를 설정해야 하는가? 객관적으로 검증 가능한 출력이 있는 스킬(파일 변환, 데이터 추출, 코드 생성, 고정된 워크플로우 단계)은 테스트 케이스의 혜택을 받는다. 주관적인 출력이 있는 스킬(글쓰기 스타일, 예술)은 종종 필요하지 않다. 스킬 유형에 따라 적절한 기본값을 제안하되, 사용자가 결정하게 한다.

### 인터뷰 및 리서치

엣지 케이스, 입출력 형식, 예시 파일, 성공 기준, 의존성에 대한 질문을 선제적으로 한다. 이 부분이 정리될 때까지 테스트 프롬프트 작성을 기다린다.

사용 가능한 MCP를 확인한다 — 리서치에 유용하면 (문서 검색, 유사 스킬 찾기, 모범 사례 조회), 서브에이전트가 있으면 병렬로 리서치한다. 맥락을 준비하여 사용자의 부담을 줄인다.

### SKILL.md 작성

사용자 인터뷰를 바탕으로 다음 구성요소를 채운다:

- **name**: 스킬 식별자
- **description**: 언제 트리거할지, 무엇을 하는지. 이것이 주요 트리거 메커니즘이다 — 스킬이 하는 것과 언제 사용하는지에 대한 구체적인 맥락을 모두 포함한다. 모든 "언제 사용할지" 정보는 본문이 아닌 여기에 넣는다. 참고: 현재 Claude는 스킬을 "과소 트리거"하는 경향이 있다 — 유용할 때 사용하지 않는다. 이를 극복하기 위해, 스킬 설명을 약간 "강하게" 작성한다. 예를 들어 "내부 Anthropic 데이터를 표시하는 간단하고 빠른 대시보드를 만드는 방법" 대신 "내부 Anthropic 데이터를 표시하는 간단하고 빠른 대시보드를 만드는 방법. 사용자가 대시보드, 데이터 시각화, 내부 지표를 언급하거나 회사 데이터를 표시하고 싶다면 - 명시적으로 '대시보드'를 요청하지 않더라도 - 반드시 이 스킬을 사용한다"라고 작성한다.
- **compatibility**: 필요한 도구, 의존성 (선택사항, 거의 필요 없음)
- **나머지 스킬 내용 :)**

### 스킬 작성 가이드

#### 스킬의 구조

```
skill-name/
├── SKILL.md (필수)
│   ├── YAML frontmatter (name, description 필수)
│   └── 마크다운 지침
└── 번들 리소스 (선택사항)
    ├── scripts/    - 결정론적/반복적 태스크를 위한 실행 가능한 코드
    ├── references/ - 필요에 따라 맥락으로 로드되는 문서
    └── assets/     - 출력에 사용되는 파일 (템플릿, 아이콘, 폰트)
```

#### 점진적 공개 (Progressive Disclosure)

스킬은 3단계 로딩 시스템을 사용한다:
1. **메타데이터** (name + description) - 항상 맥락에 포함 (~100 단어)
2. **SKILL.md 본문** - 스킬이 트리거될 때마다 맥락에 포함 (이상적으로 500줄 미만)
3. **번들 리소스** - 필요에 따라 (무제한, 스크립트는 로드 없이 실행 가능)

이 단어 수는 대략적이며 필요하면 더 길게 작성해도 된다.

**핵심 패턴:**
- SKILL.md를 500줄 미만으로 유지한다. 이 한계에 가까워지면 명확한 포인터와 함께 추가 계층 구조를 추가한다.
- SKILL.md에서 파일을 명확히 참조하고 언제 읽어야 하는지 안내한다.
- 큰 참조 파일(300줄 이상)에는 목차를 포함한다.

**도메인 구성**: 스킬이 여러 도메인/프레임워크를 지원하면 변형별로 구성한다:
```
cloud-deploy/
├── SKILL.md (워크플로우 + 선택)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude는 관련 참조 파일만 읽는다.

#### 놀라움 없음의 원칙

당연한 말이지만, 스킬은 악성 코드, 취약점 코드, 또는 시스템 보안을 손상시킬 수 있는 내용을 포함해서는 안 된다. 스킬의 내용은 설명된 의도와 달라 사용자를 놀라게 해서는 안 된다. 오해의 소지가 있는 스킬이나 무단 접근, 데이터 유출, 기타 악의적인 활동을 용이하게 하는 스킬 생성 요청을 따르지 않는다. "XYZ로 롤플레이하기" 같은 것은 OK이다.

#### 작성 패턴

지침에서는 명령형을 사용한다.

**출력 형식 정의** - 다음과 같이 할 수 있다:
```markdown
## 리포트 구조
반드시 이 정확한 템플릿을 사용한다:
# [제목]
## 요약
## 주요 발견사항
## 권고사항
```

**예시 패턴** - 예시를 포함하면 유용하다. 다음과 같이 형식화할 수 있다:
```markdown
## 커밋 메시지 형식
**예시 1:**
입력: JWT 토큰으로 사용자 인증 추가
출력: feat(auth): JWT 기반 인증 구현
```

### 작성 스타일

강압적인 MUST 대신, 중요한 이유를 모델에게 설명하려고 노력한다. 이론적 마음(theory of mind)을 사용하고 스킬을 일반적으로 만들되 특정 예시에 과도하게 좁혀지지 않도록 한다. 초안을 작성한 후 새로운 눈으로 보고 개선한다.

### 테스트 케이스

스킬 초안 작성 후, 2-3개의 현실적인 테스트 프롬프트를 생각해낸다 — 실제 사용자가 실제로 말할 것 같은 종류. 사용자에게 공유한다: [이 정확한 말을 사용할 필요는 없다] "시도해보고 싶은 몇 가지 테스트 케이스입니다. 맞아 보이나요, 아니면 더 추가하시겠어요?" 그런 다음 실행한다.

테스트 케이스를 `evals/evals.json`에 저장한다. 아직 assertion을 작성하지 않는다 — 프롬프트만. assertion은 다음 단계에서 실행이 진행되는 동안 작성한다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자의 태스크 프롬프트",
      "expected_output": "예상 결과 설명",
      "files": []
    }
  ]
}
```

전체 스키마(assertions 필드 포함)는 `references/schemas.md`를 참조한다.

## 테스트 케이스 실행 및 평가

이 섹션은 하나의 연속된 순서이다 — 중간에 멈추지 않는다. `/skill-test`나 다른 테스트 스킬을 사용하지 않는다.

결과를 스킬 디렉토리의 형제 디렉토리인 `<skill-name>-workspace/`에 넣는다. 워크스페이스 내에서 반복 횟수별로 결과를 구성한다(`iteration-1/`, `iteration-2/` 등), 그 안에서 각 테스트 케이스는 디렉토리를 갖는다 (`eval-0/`, `eval-1/` 등). 이 모든 것을 미리 만들지 않는다 — 진행하면서 디렉토리를 만든다.

### 1단계: 같은 턴에서 모든 실행 생성 (with-skill AND baseline)

각 테스트 케이스에 대해, 같은 턴에서 두 서브에이전트를 생성한다 — 스킬 있는 것과 없는 것. 이것이 중요하다: with-skill 실행을 먼저 생성하고 나중에 baseline으로 돌아오지 않는다. 모든 것이 비슷한 시간에 완료되도록 한 번에 모두 실행한다.

**With-skill 실행:**

```
이 태스크를 실행한다:
- 스킬 경로: <path-to-skill>
- 태스크: <eval prompt>
- 입력 파일: <eval files if any, or "none">
- 출력 저장 위치: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 저장할 출력: <사용자가 중요하게 여기는 것 — 예: ".docx 파일", "최종 CSV">
```

**Baseline 실행** (동일한 프롬프트, baseline은 맥락에 따라 다름):
- **새 스킬 만들기**: 스킬 없음. 동일한 프롬프트, 스킬 경로 없음, `without_skill/outputs/`에 저장.
- **기존 스킬 개선**: 이전 버전. 편집 전에 스킬을 스냅샷 (`cp -r <skill-path> <workspace>/skill-snapshot/`), 그런 다음 baseline 서브에이전트가 스냅샷을 가리키게 한다. `old_skill/outputs/`에 저장.

각 테스트 케이스에 대해 `eval_metadata.json`을 작성한다 (assertion은 지금은 비어있어도 됨). 무엇을 테스트하는지에 기반한 설명적인 이름을 각 평가에 부여한다 — "eval-0"이 아닌 것으로. 이 이름을 디렉토리에도 사용한다. 이번 반복이 새로운 또는 수정된 eval 프롬프트를 사용하면, 각 새 eval 디렉토리에 이 파일들을 만든다 — 이전 반복에서 이어진다고 가정하지 않는다.

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "사용자의 태스크 프롬프트",
  "assertions": []
}
```

### 2단계: 실행이 진행되는 동안 assertion 초안 작성

실행이 완료되길 기다리지 않는다 — 이 시간을 생산적으로 활용할 수 있다. 각 테스트 케이스에 대한 정량적 assertion을 초안 작성하고 사용자에게 설명한다. `evals/evals.json`에 assertion이 이미 존재하면 검토하고 무엇을 확인하는지 설명한다.

좋은 assertion은 객관적으로 검증 가능하고 설명적인 이름을 갖는다 — 벤치마크 뷰어에서 명확하게 읽혀서 결과를 보는 누군가가 각각이 무엇을 확인하는지 즉시 이해할 수 있어야 한다. 주관적인 스킬(글쓰기 스타일, 디자인 품질)은 정성적으로 평가하는 것이 더 낫다 — 인간의 판단이 필요한 것에 억지로 assertion을 적용하지 않는다.

초안이 완성되면 `eval_metadata.json` 파일과 `evals/evals.json`을 assertion으로 업데이트한다. 또한 사용자에게 뷰어에서 무엇을 보게 될지 설명한다 — 정성적 출력과 정량적 벤치마크 모두.

### 3단계: 실행이 완료되면 타이밍 데이터 캡처

각 서브에이전트 태스크가 완료되면, `total_tokens`와 `duration_ms`가 포함된 알림을 받는다. 이 데이터를 즉시 실행 디렉토리의 `timing.json`에 저장한다:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

이것이 이 데이터를 캡처할 수 있는 유일한 기회이다 — 태스크 알림을 통해 전달되며 다른 곳에 저장되지 않는다. 일괄 처리하려 하지 말고 알림이 도착할 때마다 처리한다.

### 4단계: 채점, 집계, 뷰어 실행

모든 실행이 완료되면:

1. **각 실행 채점** — 채점기 서브에이전트를 생성하거나 (또는 인라인으로 채점) `agents/grader.md`를 읽고 각 assertion을 출력에 대해 평가한다. 결과를 각 실행 디렉토리의 `grading.json`에 저장한다. grading.json expectations 배열은 반드시 `text`, `passed`, `evidence` 필드를 사용해야 한다 (`name`/`met`/`details` 또는 다른 변형이 아닌) — 뷰어가 이 정확한 필드명에 의존한다. 프로그래밍 방식으로 확인할 수 있는 assertion의 경우, 눈으로 확인하는 것보다 스크립트를 작성하고 실행한다 — 스크립트가 더 빠르고, 신뢰할 수 있으며, 반복에 걸쳐 재사용할 수 있다.

2. **벤치마크로 집계** — skill-creator 디렉토리에서 집계 스크립트를 실행한다:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   이것은 각 설정에 대한 pass_rate, time, tokens와 평균 ± 표준편차 및 delta가 있는 `benchmark.json`과 `benchmark.md`를 생성한다. benchmark.json을 수동으로 생성하는 경우, 뷰어가 예상하는 정확한 스키마에 대해 `references/schemas.md`를 참조한다.
   with_skill 버전을 baseline 상대방보다 먼저 넣는다.

3. **분석가 패스 수행** — 벤치마크 데이터를 읽고 집계 통계가 숨길 수 있는 패턴을 표면화한다. 찾아야 할 것에 대해 `agents/analyzer.md` ("벤치마크 결과 분석" 섹션)를 참조한다 — 스킬 가치를 구별하지 못하는 assertion(비구별적), 분산이 높은 평가(불안정 가능성), time/token 트레이드오프 같은 것들.

4. 정성적 출력과 정량적 데이터로 **뷰어 실행**:
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   반복 2 이상에서는 `--previous-workspace <workspace>/iteration-<N-1>`도 전달한다.

   **Cowork / headless 환경:** `webbrowser.open()`이 사용 불가능하거나 환경에 디스플레이가 없으면, 서버를 시작하는 대신 `--static <output_path>`를 사용하여 독립형 HTML 파일을 작성한다. 사용자가 "모든 리뷰 제출"을 클릭하면 피드백이 `feedback.json` 파일로 다운로드된다. 다운로드 후, 다음 반복이 받을 수 있도록 `feedback.json`을 워크스페이스 디렉토리로 복사한다.

   참고: 뷰어 생성에는 generate_review.py를 사용한다. 커스텀 HTML을 작성할 필요가 없다.

5. **사용자에게 알림**: 다음과 같이 말한다: "브라우저에서 결과를 열었습니다. 탭이 두 개 있습니다 — '출력'은 각 테스트 케이스를 클릭하고 피드백을 남길 수 있고, '벤치마크'는 정량적 비교를 보여줍니다. 완료하면 돌아와서 알려주세요."

### 뷰어에서 사용자가 보는 것

"출력" 탭은 한 번에 하나의 테스트 케이스를 보여준다:
- **프롬프트**: 주어진 태스크
- **출력**: 스킬이 생성한 파일, 가능하면 인라인으로 렌더링
- **이전 출력** (반복 2 이상): 마지막 반복의 출력을 접힌 섹션으로 표시
- **공식 채점** (채점을 실행한 경우): assertion 통과/실패를 보여주는 접힌 섹션
- **피드백**: 입력하면서 자동 저장되는 텍스트박스
- **이전 피드백** (반복 2 이상): 마지막 번의 댓글, 텍스트박스 아래에 표시

"벤치마크" 탭은 통계 요약을 보여준다: 각 설정에 대한 통과율, 타이밍, 토큰 사용량, 평가별 분석 및 분석가 관찰 사항.

탐색은 이전/다음 버튼 또는 화살표 키로 한다. 완료하면 "모든 리뷰 제출"을 클릭하여 모든 피드백을 `feedback.json`에 저장한다.

### 5단계: 피드백 읽기

사용자가 완료했다고 하면, `feedback.json`을 읽는다:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "차트에 축 레이블이 없음", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "완벽해, 좋아", "timestamp": "..."}
  ],
  "status": "complete"
}
```

빈 피드백은 사용자가 괜찮다고 생각했다는 의미이다. 사용자가 구체적인 불만을 가진 테스트 케이스에 집중한다.

완료되면 뷰어 서버를 종료한다:

```bash
kill $VIEWER_PID 2>/dev/null
```

## 스킬 개선

이것이 루프의 핵심이다. 테스트 케이스를 실행하고, 사용자가 결과를 검토했으며, 이제 피드백을 바탕으로 스킬을 더 낫게 만들어야 한다.

### 개선을 생각하는 방법

1. **피드백에서 일반화한다.** 여기서 일어나는 큰 그림은 수백만 번 (어쩌면 말 그대로, 어쩌면 그 이상) 다양한 프롬프트에 걸쳐 사용할 수 있는 스킬을 만들려는 것이다. 여기서 사용자와 함께 몇 가지 예시만 반복하는 것은 더 빠르게 진행하기 위해서이다. 사용자는 이 예시들을 잘 알고 새로운 출력을 빠르게 평가할 수 있다. 하지만 작성 중인 스킬이 그 예시들에만 적용된다면 쓸모없다. 자잘하고 과적합된 변경이나 지나치게 제한적인 MUST 대신, 완고한 문제가 있다면 다른 은유를 사용하거나 다른 작업 패턴을 권장해보는 것을 시도한다. 시도하는 데 비용이 적게 들고 어쩌면 좋은 것을 발견할 수도 있다.

2. **프롬프트를 간결하게 유지한다.** 효과가 없는 것들을 제거한다. 실행 기록을 읽는다, 최종 출력만이 아닌 — 스킬이 모델이 비생산적인 일을 하는 데 많은 시간을 낭비하게 만드는 것처럼 보인다면, 그 부분을 제거하고 어떻게 되는지 시도할 수 있다.

3. **이유를 설명한다.** 요청하는 모든 것 뒤에 **왜**를 설명하려고 노력한다. 오늘날의 LLM은 *똑똑하다*. 좋은 이론적 마음을 갖고 있으며 좋은 환경이 주어지면 단순한 지침을 넘어 실제로 일을 이룰 수 있다. 사용자의 피드백이 간결하거나 좌절스럽더라도, 태스크가 무엇인지, 왜 사용자가 그것을 작성했는지, 실제로 무엇을 원하는지 이해하려 노력한다. 그런 다음 이 이해를 지침에 담는다. ALWAYS나 NEVER를 대문자로 쓰고 있다면, 또는 매우 경직된 구조를 사용하고 있다면, 그것은 황색 경고이다 — 가능하면 재구성하여 왜 요청하는 것이 중요한지 이유를 설명한다. 그것이 더 인도적이고, 강력하며, 효과적인 접근 방식이다.

4. **테스트 케이스 전반에 걸쳐 반복되는 작업을 찾는다.** 테스트 실행의 기록을 읽고 모든 서브에이전트가 비슷한 헬퍼 스크립트를 독립적으로 작성했는지 또는 동일한 다단계 접근 방식을 취했는지 확인한다. 3개의 테스트 케이스가 모두 서브에이전트가 `create_docx.py`나 `build_chart.py`를 작성하는 결과를 초래했다면, 스킬이 그 스크립트를 번들해야 한다는 강력한 신호이다. 한 번 작성하고, `scripts/`에 넣고, 스킬이 사용하도록 지시한다. 이것은 모든 미래 호출이 바퀴를 다시 발명하는 것을 절약한다.

이 태스크는 꽤 중요하며 (여기서 연간 수십억 달러의 경제적 가치를 만들려고 노력하고 있다!) 생각하는 시간이 병목이 아니다. 시간을 내서 정말로 충분히 생각한다. 수정 초안을 작성하고 새로운 눈으로 보고 개선하기를 제안한다. 정말로 사용자의 머릿속에 들어가서 원하는 것과 필요한 것을 이해하려고 노력한다.

### 반복 루프

스킬을 개선한 후:

1. 스킬에 개선 사항을 적용한다.
2. 새로운 `iteration-<N+1>/` 디렉토리에 모든 테스트 케이스를 다시 실행한다, baseline 실행 포함. 새 스킬을 만드는 경우, baseline은 항상 `without_skill`(스킬 없음)이다 — 반복 전반에 걸쳐 동일하게 유지된다. 기존 스킬을 개선하는 경우, baseline으로 무엇이 의미 있는지 판단한다: 사용자가 가져온 원래 버전 또는 이전 반복.
3. `--previous-workspace`가 이전 반복을 가리키도록 뷰어를 실행한다.
4. 사용자가 검토하고 완료했다고 할 때까지 기다린다.
5. 새로운 피드백을 읽고, 다시 개선하고, 반복한다.

다음 중 하나가 될 때까지 계속한다:
- 사용자가 만족한다고 할 때까지
- 피드백이 모두 비어있을 때까지 (모든 것이 좋아 보임)
- 의미 있는 진전을 하지 못할 때까지

## 고급: 블라인드 비교

두 버전의 스킬 간에 더 엄격한 비교를 원하는 상황 (예: "새 버전이 실제로 더 나은가?")을 위한 블라인드 비교 시스템이 있다. `agents/comparator.md`와 `agents/analyzer.md`에서 자세한 내용을 확인한다. 기본 아이디어는: 어느 쪽인지 알려주지 않고 독립적인 에이전트에게 두 출력을 제공하여 품질을 판단하게 한다. 그런 다음 승자가 왜 이겼는지 분석한다.

이것은 선택사항이며 서브에이전트가 필요하고 대부분의 사용자는 필요하지 않다. 인간 리뷰 루프가 보통 충분하다.

## 설명 최적화

SKILL.md frontmatter의 description 필드는 Claude가 스킬을 호출할지 결정하는 주요 메커니즘이다. 스킬을 만들거나 개선한 후, 더 나은 트리거 정확도를 위해 설명을 최적화하도록 제안한다.

### 1단계: 트리거 평가 쿼리 생성

20개의 평가 쿼리를 생성한다 — should-trigger와 should-not-trigger의 혼합. JSON으로 저장한다:

```json
[
  {"query": "사용자 프롬프트", "should_trigger": true},
  {"query": "다른 프롬프트", "should_trigger": false}
]
```

쿼리는 실제적이어야 하고 Claude Code나 Claude.ai 사용자가 실제로 입력할 것이어야 한다. 추상적인 요청이 아닌, 구체적이고 자세한 요청: 파일 경로, 직업이나 상황에 대한 개인적 맥락, 열 이름과 값, 회사 이름, URL. 약간의 배경. 일부는 소문자이거나 약어, 오타, 구어체를 포함할 수 있다. 다양한 길이를 사용하고, 명백한 것보다 엣지 케이스에 집중한다 (사용자가 검토할 기회가 있을 것이다).

나쁜 예: `"이 데이터를 서식화해줘"`, `"PDF에서 텍스트 추출해줘"`, `"차트 만들어줘"`

좋은 예: `"ok so 내 상사가 방금 이 xlsx 파일을 보냈어 (다운로드 폴더에 있는데, 'Q4 sales final FINAL v2.xlsx' 같은 이름이야) 이익 마진을 백분율로 보여주는 열을 추가하길 원해. 수익은 C열에 있고 비용은 아마 D열인 것 같아"`

**should-trigger 쿼리** (8-10개): 커버리지를 생각한다. 동일한 의도에 대한 다양한 표현을 원한다 — 일부는 공식적이고 일부는 비공식적. 사용자가 스킬이나 파일 유형을 명시적으로 명명하지 않지만 명확히 필요한 경우를 포함한다. 일반적이지 않은 사용 사례와 이 스킬이 다른 것과 경쟁하지만 이겨야 하는 경우를 추가한다.

**should-not-trigger 쿼리** (8-10개): 가장 가치 있는 것은 근사 미스(near-misses)이다 — 스킬과 키워드나 개념을 공유하지만 실제로 다른 것이 필요한 쿼리. 순진한 키워드 매칭이 트리거하지만 실제로는 트리거하지 않아야 하는 인접 도메인, 모호한 표현, 스킬이 하는 것을 건드리지만 다른 도구가 더 적절한 맥락의 경우를 생각한다.

핵심은 should-not-trigger 쿼리를 명백하게 무관하게 만들지 않는 것이다. PDF 스킬에 대한 부정적 테스트로 "피보나치 함수 작성해줘"는 너무 쉽다 — 아무것도 테스트하지 않는다. 부정 케이스는 진정으로 까다로워야 한다.

### 2단계: 사용자와 검토

HTML 템플릿을 사용하여 사용자에게 평가 세트를 검토를 위해 제시한다:

1. `assets/eval_review.html`에서 템플릿을 읽는다.
2. 플레이스홀더를 교체한다:
   - `__EVAL_DATA_PLACEHOLDER__` → 평가 항목의 JSON 배열 (따옴표 없이 — JS 변수 할당)
   - `__SKILL_NAME_PLACEHOLDER__` → 스킬 이름
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 스킬의 현재 설명
3. 임시 파일에 작성 (예: `/tmp/eval_review_<skill-name>.html`)하고 열기: `open /tmp/eval_review_<skill-name>.html`
4. 사용자는 쿼리를 편집하고, should-trigger를 전환하고, 항목을 추가/제거하고, "평가 세트 내보내기"를 클릭할 수 있다.
5. 파일이 `~/Downloads/eval_set.json`으로 다운로드된다 — 여러 개가 있는 경우 가장 최신 버전을 위해 Downloads 폴더를 확인한다 (예: `eval_set (1).json`).

이 단계가 중요하다 — 나쁜 평가 쿼리는 나쁜 설명을 초래한다.

### 3단계: 최적화 루프 실행

사용자에게 말한다: "이것은 시간이 걸릴 것입니다 — 백그라운드에서 최적화 루프를 실행하고 주기적으로 확인하겠습니다."

평가 세트를 워크스페이스에 저장하고, 백그라운드에서 실행한다:

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

현재 세션을 구동하는 시스템 프롬프트에서 모델 ID를 사용한다. 그러면 트리거 테스트가 사용자가 실제로 경험하는 것과 일치한다.

실행되는 동안, 주기적으로 출력을 확인하여 사용자에게 어떤 반복인지와 점수가 어떤지 업데이트를 제공한다.

이것은 전체 최적화 루프를 자동으로 처리한다. 평가 세트를 60% 훈련과 40% 보류 테스트로 분할하고, 현재 설명을 평가하며 (각 쿼리를 3번 실행하여 신뢰할 수 있는 트리거율 획득), 실패한 것을 기반으로 개선을 제안하기 위해 Claude를 호출한다. 훈련 및 테스트 모두에서 각 새로운 설명을 재평가하고, 최대 5번 반복한다. 완료되면 과적합을 피하기 위해 훈련 점수가 아닌 테스트 점수로 선택된 `best_description`과 함께 JSON을 반환한다.

### 스킬 트리거 작동 방식

트리거 메커니즘을 이해하면 더 나은 평가 쿼리를 설계하는 데 도움이 된다. 스킬은 이름 + 설명과 함께 Claude의 `available_skills` 목록에 나타나며, Claude는 그 설명을 기반으로 스킬을 참조할지 결정한다. 중요한 것은 Claude가 스스로 쉽게 처리할 수 없는 태스크에 대해서만 스킬을 참조한다는 것이다 — "이 PDF를 읽어줘"와 같은 간단하고 단일 단계 쿼리는 설명이 완벽하게 일치하더라도 스킬을 트리거하지 않을 수 있다. 복잡하고 다단계이거나 특수화된 쿼리는 설명이 일치할 때 스킬을 안정적으로 트리거한다.

이것은 평가 쿼리가 Claude가 실제로 스킬을 참조하여 혜택을 받을 만큼 충분히 실질적이어야 함을 의미한다. "파일 X를 읽어줘"와 같은 간단한 쿼리는 설명 품질에 관계없이 스킬을 트리거하지 않으므로 좋지 않은 테스트 케이스이다.

### 4단계: 결과 적용

JSON 출력에서 `best_description`을 가져와 스킬의 SKILL.md frontmatter를 업데이트한다. 사용자에게 변경 전후를 보여주고 점수를 보고한다.

---

### 패키지 및 제시 (`present_files` 도구가 사용 가능한 경우에만)

`present_files` 도구에 접근할 수 있는지 확인한다. 없으면 이 단계를 건너뛴다. 있으면 스킬을 패키지화하고 사용자에게 .skill 파일을 제시한다:

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

패키지화 후, 사용자가 설치할 수 있도록 결과 `.skill` 파일 경로를 알려준다.

## Claude.ai 특화 지침

Claude.ai에서는 핵심 워크플로우가 동일하지만 (초안 → 테스트 → 검토 → 개선 → 반복), Claude.ai에는 서브에이전트가 없으므로 일부 메커니즘이 변경된다. 다음을 조정한다:

**테스트 케이스 실행**: 서브에이전트 없음은 병렬 실행 없음을 의미한다. 각 테스트 케이스에 대해 스킬의 SKILL.md를 읽은 후, 그 지침에 따라 테스트 프롬프트를 직접 수행한다. 하나씩 진행한다. 이것은 독립적인 서브에이전트보다 덜 엄격하지만 (스킬을 작성하고 실행하므로 전체 맥락이 있음), 유용한 검토이다 — 그리고 인간 검토 단계가 보완한다. Baseline 실행은 건너뛴다 — 스킬을 사용하여 요청대로 태스크를 완료하는 것만 한다.

**결과 검토**: 브라우저를 열 수 없으면 (예: Claude.ai의 VM에 디스플레이가 없거나 원격 서버에 있는 경우), 브라우저 검토기를 완전히 건너뛴다. 대신 대화에서 직접 결과를 제시한다. 각 테스트 케이스에 대해 프롬프트와 출력을 보여준다. 출력이 사용자가 봐야 하는 파일이면 (예: .docx나 .xlsx), 파일시스템에 저장하고 다운로드하고 검사할 수 있도록 위치를 알려준다. 인라인으로 피드백을 요청한다: "이것이 어떻게 보이나요? 변경하고 싶은 것이 있나요?"

**벤치마킹**: 정량적 벤치마킹은 건너뛴다 — 서브에이전트 없이는 의미 있는 baseline 비교에 의존한다. 사용자의 정성적 피드백에 집중한다.

**반복 루프**: 이전과 동일하다 — 스킬을 개선하고, 테스트 케이스를 다시 실행하고, 피드백을 요청한다 — 중간에 브라우저 검토기 없이. 파일시스템이 있으면 반복 디렉토리에 결과를 구성할 수 있다.

**설명 최적화**: 이 섹션은 Claude Code에서만 사용 가능한 `claude` CLI 도구 (특히 `claude -p`)가 필요하다. Claude.ai에서는 건너뛴다.

**블라인드 비교**: 서브에이전트가 필요하다. 건너뛴다.

**패키지화**: `package_skill.py` 스크립트는 Python과 파일시스템이 있는 어디서나 작동한다. Claude.ai에서 실행할 수 있으며 사용자는 결과 `.skill` 파일을 다운로드할 수 있다.

**기존 스킬 업데이트**: 사용자가 새 스킬 생성이 아닌 기존 스킬 업데이트를 요청할 수 있다. 이 경우:
- **원래 이름을 유지한다.** 스킬의 디렉토리 이름과 `name` frontmatter 필드를 확인한다 -- 변경 없이 사용한다. 예: 설치된 스킬이 `research-helper`이면, `research-helper-v2`가 아닌 `research-helper.skill`을 출력한다.
- **편집 전에 쓰기 가능한 위치로 복사한다.** 설치된 스킬 경로가 읽기 전용일 수 있다. `/tmp/skill-name/`으로 복사하고, 거기서 편집하고, 복사본에서 패키지화한다.
- **수동으로 패키지화하는 경우, 먼저 `/tmp/`에서 준비한 다음** 출력 디렉토리로 복사한다 -- 직접 쓰기는 권한 때문에 실패할 수 있다.

## Cowork 특화 지침

Cowork에 있다면 알아야 할 주요 사항:

- 서브에이전트가 있으므로 메인 워크플로우(테스트 케이스를 병렬로 생성, baseline 실행, 채점 등)가 모두 작동한다. (타임아웃 문제가 심각하면 테스트 프롬프트를 병렬 대신 직렬로 실행하는 것도 OK이다.)
- 브라우저나 디스플레이가 없으므로 eval 뷰어를 생성할 때, 서버를 시작하는 대신 `--static <output_path>`를 사용하여 독립형 HTML 파일을 작성한다. 그런 다음 사용자가 브라우저에서 HTML을 열 수 있는 링크를 제공한다.
- 어떤 이유에서인지 Cowork 설정은 테스트 실행 후 Claude가 eval 뷰어를 생성하는 것을 꺼리게 만드는 것 같다. 강조하자면: Cowork에 있든 Claude Code에 있든, 테스트 실행 후에는 반드시 `generate_review.py` (커스텀 html 코드를 작성하는 것이 아닌)를 사용하여 eval 뷰어를 생성하여 인간이 검토할 수 있게 한다. 미리 사과하지만 대문자로 쓰겠다: 스스로 입력을 평가하기 *전에* EVAL 뷰어를 생성한다. 최대한 빨리 인간이 볼 수 있게 해야 한다!
- 피드백은 다르게 작동한다: 실행 중인 서버가 없으므로, 뷰어의 "모든 리뷰 제출" 버튼이 `feedback.json`을 파일로 다운로드한다. 그런 다음 거기서 읽을 수 있다 (먼저 접근을 요청해야 할 수 있다).
- 패키지화는 작동한다 — `package_skill.py`는 Python과 파일시스템만 필요하다.
- 설명 최적화 (`run_loop.py` / `run_eval.py`)는 브라우저가 아닌 서브프로세스를 통해 `claude -p`를 사용하므로 Cowork에서도 잘 작동해야 한다. 스킬이 완전히 완성되고 사용자가 만족할 때까지 저장한다.
- **기존 스킬 업데이트**: 사용자가 새 스킬 생성이 아닌 기존 스킬 업데이트를 요청할 수 있다. 위의 claude.ai 섹션에서 업데이트 지침을 따른다.
