# AI 지침 압축 설계론

> 사람에게 읽기 쉬운 코드가 AI에게도 이해하기 쉬운 코드가 아닐 수 있다.

---

## 1. AI에게 좋은 지침의 조건

### 간결성

- 미사여구 제거 — 이해해야 할 맥락이 적을수록 AI 성능이 높아진다
- 장황한 설명은 오해를 유발한다
- 사람을 위한 친절한 주석이 AI에게는 노이즈다

### 구조성

- 리스트, 표, 키-값 쌍 등 AI가 파싱하기 쉬운 형식을 사용한다
- 구조가 없으면 AI가 스스로 구조를 유추하며 토큰을 낭비하고 의도와 다르게 해석할 수 있다

### 구체성

- 주어와 목적어가 분명해야 한다
- 단어 선택은 문서 전체에서 일관성을 유지한다

---

## 2. 직관 → 구체의 파이프라인

사람의 생각은 다음 순서로 변화한다.

```
직관적  →  추상적  →  구체적
```

AI를 잘 쓰는 것의 핵심은 이 전환을 빠르게 만드는 것이다.

### 반복 추상화 사이클

```
1. 인간: 직관적 생각을 그대로 AI에게 전달한다
        ↓
2. AI:   추상화된 표현으로 리턴한다
        ↓
3. 인간: 리턴된 표현을 검토하고 수정 방향을 전달한다
        ↓
4. 반복 → 점진적으로 구체화된다
```

이 과정을 생략하면 AI가 잘못 이해한 채로 작업을 진행해 이상한 아웃풋이 나온다.
`instruction-transformer` 스킬이 이 사이클을 담당할 수 있다.

---

## 3. 하네스 엔지니어링의 한계

현재 하네스 엔지니어링(스킬, 커맨드, 훅, MCP 등)은 다음의 한계가 있다.

| 한계 | 내용 |
|------|------|
| 경직성 | 특정 애플리케이션 제작에 최적화된 구조 |
| 비범용성 | 작업마다 새로 세팅하는 경우가 많다 |
| 재사용성 부족 | 이전 하네스가 다음 작업에 그대로 쓰이지 못한다 |

### 범용 하네스의 방향

클린 아키텍처가 코딩의 모든 "만들기 과정"에 동일한 원칙을 적용하듯,
서류, 제품, 코드 — 무언가를 만드는 모든 과정(Building)에 공통으로 적용되는
**개념적 분할 원칙**이 필요하다.

## 4. AI 명령 압축 (AI Regex)

지침을 작성하는 것 자체가 병목이다.
정규식처럼 AI 명령을 기호로 압축하면 지침 작성 비용을 줄일 수 있다.

### 개념 스케치

```
[ AI정규식 덩어리 ]  자연어  [ AI정규식 덩어리 ]
```

- `!` — 반드시 실행되어야 하는 항목
- (추가 기호는 설계 중)

### 기대 효과

| 현재 | 압축 후 |
|------|---------|
| 긴 자연어 지시문 | 기호 + 핵심 단어 조합 |
| 토큰 낭비 | 토큰 절약 |
| 해석 편차 발생 | 명확한 의미 고정 |

### instruction-transformer 스킬과의 연계

```
1. 인간: AI Regex 표현으로 지침 초안 작성
        ↓
2. instruction-transformer: 일반 자연어로 변환하여 리턴
        ↓
3. 인간: 검토 후 확정
        ↓
4. 확정된 지침을 실제 AI 지시문으로 사용
```

압축 표현으로 빠르게 의도를 전달하고, 스킬이 전개(expand)하는 구조다.

## 5. 범용 하네스 셋업 비전

하네스 세팅도 명령어로 간단히 구성될 수 있어야 한다.

```
# 예시 (아이디어 단계)
!init harness --type=document
!add skill instruction-transformer
!add hook pre-commit linter
```

- 작업 유형(문서, 제품, 코드 등)에 따라 적합한 스킬·훅이 자동으로 선택된다
- 범용 원칙 위에서 작업별 하네스가 파생되는 구조
