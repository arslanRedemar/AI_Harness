## 1. `cps-completeness-checker`

**역할**: PRD 착수 가능 여부를 판단한다. 미흡한 항목은 다음 미팅 안건으로 전환한다.

**입력**
```
- CPS 누적 문서 전체 (모든 미팅의 Context / Problem / Solution)
- 프로젝트 유형 힌트 (앱 개발 / 데이터 파이프라인 / 운영 자동화 등)
```

**처리 단계**
```
1. Context 평가
   - 사용자 유형이 2개 이상 식별되었는가?
   - 핵심 비즈니스 흐름(Happy Path)이 서술되었는가?
   - 기술적 제약 조건이 1개 이상 명시되었는가?
   - 규모 수치(트래픽·사용자 수·데이터량 등)가 1개 이상 존재하는가?
   - 외부 시스템 연동 범위가 식별되었는가?

2. Problem 평가
   - 핵심 문제가 "현재 상태 → 목표 상태" 형식으로 정의되었는가?
   - 측정 가능한 Pain Point가 1개 이상 있는가?
   - 사용자 유형별 문제가 분리되었는가?

3. Solution 평가
   - 핵심 기술 결정 중 확정된 것이 보류 중인 것보다 많은가?
   - 기능 목록의 윤곽이 최소 3개 이상 존재하는가?
   - 1차 스코프와 이후 스코프가 한 번이라도 논의되었는가?

4. 항목별 상태 분류: OK / PARTIAL / MISSING
5. 완성도 점수 산출: (OK × 1 + PARTIAL × 0.5) / 전체 항목 수
6. 임계값 판단: 점수 ≥ 0.7이면 PRD 착수 권장, 미만이면 보완 안건 생성
```

**출력 예시**
```
[CPS 완성도 리포트]

Context
  ✅ 사용자 유형 식별: 기사·고객·운영팀 (3종)
  ✅ 핵심 비즈니스 흐름: 주문 접수 → 기사 배정 → 배달 완료
  ✅ 기술 제약: React Native, Node.js, PostgreSQL 확정
  ✅ 규모 수치: 피크 분당 200건, 기사 150명
  ⚠️ 외부 연동: PG사 미결 (결제 범위 TBD)

Problem
  ✅ 핵심 문제: 수동 주문 → 앱 전환 (디지털화)
  ✅ 측정 가능 Pain Point: 피크타임 응답 지연
  ⚠️ 사용자별 분리: 운영팀 Pain Point 미기술

Solution
  ✅ 확정 결정 5건 > 보류 2건
  ✅ 기능 윤곽: 주문·위치추적·어드민·알림 (4개)
  ❌ 1차 스코프 경계: 명시적 합의 없음

완성도 점수: 8.5 / 13 = 0.65

판정: ⚠️ PRD 착수 전 보완 권장
다음 미팅 안건 (3건):
  1. 결제 범위 확정 (1차 스코프 포함 여부)
  2. 운영팀 Pain Point 인터뷰
  3. 1차 스코프 vs 2차 스코프 경계 합의
```

## 2. `persona-extractor`

**역할**: CPS Context에서 사용자 유형별 페르소나를 추출하여 PRD 페르소나 섹션을 구성한다.

**입력**
```
- CPS Context 누적 (전 미팅)
- 고객사 업종 힌트
```

**처리 단계**
```
1. Context에서 언급된 모든 사용자 역할 추출
2. 역할별로 아래 항목 취합:
   - 목표: 이 사람이 시스템으로 달성하려는 것
   - 주요 행동: 시스템에서 반복적으로 하는 작업
   - Pain Point: 현재 겪는 불편
   - 민감 조건: 반드시 충족해야 하는 제약 (성능·접근성·권한 등)
3. 미팅에서 언급되지 않은 항목은 "(미확인)"으로 표기
4. 페르소나 카드 형식으로 출력
```

**출력 예시**
```
[페르소나 카드]

## 페르소나 A — 배달 기사 (이동우)
- 목표: 주문을 빠르게 수령하고 배달 완료 처리
- 주요 행동: 앱으로 주문 수신 → 픽업 이동 → 배달 완료 버튼 탭
- Pain Point: 위치 업데이트 지연 시 고객 항의, 주문 취소 시 공지 느림
- 민감 조건: 이동 중 한 손 조작, 저사양 기기 대응 필요

## 페르소나 B — 고객 (주문자)
- 목표: 주문 후 배달 상태를 실시간으로 파악
- 주요 행동: 주문 → 실시간 위치 확인 → 배달 완료 확인
- Pain Point: 배달 상태가 언제 바뀌는지 불명확
- 민감 조건: (미확인)

## 페르소나 C — 운영팀 관리자
- 목표: 전체 기사 현황 모니터링 + 이슈 대응
- 주요 행동: 어드민 지도에서 기사 위치 확인, 주문 현황 조회
- Pain Point: (미확인 — 다음 미팅 안건)
- 민감 조건: 비기술직, 복잡한 UI 거부감
```

## 3. `goal-metric-definer`

**역할**: CPS Problem을 "현재 상태 → 목표 상태 → KPI → 측정 방법" 체인으로 변환한다.

**입력**
```
- CPS Problem 누적 전체
- CPS Context의 규모 수치
```

**처리 단계**
```
1. Problem 항목별로 "현재 상태" 추출
2. 각 현재 상태에 대응하는 "목표 상태" 정의
   - 미팅에서 명시된 경우 → 그대로 사용
   - 암묵적인 경우 → 업종 표준 기준 제안 (출처 표기)
3. 목표 상태를 측정 가능한 KPI로 변환
   - "빠르게" → "p99 응답시간 < Xms"
   - "안 터지게" → "가용성 99.9% 이상"
   - "편리하게" → "작업 완료까지 탭 수 3회 이하" 등
4. KPI별 측정 방법과 측정 시점 정의
5. 베이스라인 존재 여부 표기 (없으면 "(측정 필요)"로 표기)
```

**출력 예시**
```
[목표 및 KPI]

| 현재 상태 | 목표 상태 | KPI | 측정 방법 | 베이스라인 |
|-----------|----------|-----|----------|-----------|
| 카카오채널 수동 주문 | 앱 주문 전환율 80% 이상 | 앱 주문 비율 | 월별 주문 채널 집계 | (측정 필요) |
| 피크타임 응답 지연 우려 | 피크타임 API p99 < 300ms | API 응답시간 p99 | APM 모니터링 | (측정 필요) |
| 기사 위치 업데이트 불명확 | 위치 갱신 주기 3초 이하 | 위치 업데이트 지연 | 클라이언트 수신 로그 | (측정 필요) |
| 수동 운영 모니터링 | 어드민에서 전체 기사 실시간 확인 | 어드민 지도 갱신 주기 | 화면 스냅샷 비교 | — |
```

## 4. `feature-distiller`

**역할**: CPS Solution 중 확정된 항목만 골라 Epic → Feature → Story 계층으로 분해한다.

**입력**
```
- CPS Solution 누적 (확정 표시된 항목만)
- 페르소나 목록 (persona-extractor 출력)
```

**처리 단계**
```
1. Solution 항목에서 확정 결정만 필터링 (보류·TBD 제외)
2. 기능 성격에 따라 Epic으로 그룹핑
   - 사용자 행동 단위: 주문, 위치추적, 알림, 결제 등
3. 각 Epic을 구체적 Feature로 분해
4. 각 Feature를 페르소나 관점의 User Story로 재서술
   - 형식: "나는 [페르소나]로서 [행동]을 통해 [목적]을 달성하고 싶다"
5. 보류 중인 Solution은 별도 "미결 기능" 목록으로 분리
```

**출력 예시**
```
[기능 계층 구조]

Epic 1: 주문 관리
  Feature 1-1: 고객 앱 주문 생성
    Story: 나는 고객으로서 앱에서 메뉴를 선택하고 주문을 넣어
           배달을 요청하고 싶다
  Feature 1-2: 기사 앱 주문 수신
    Story: 나는 기사로서 새 주문이 들어오면 즉시 알림을 받아
           수락 또는 거절하고 싶다
  Feature 1-3: 주문 상태 관리 (접수→픽업→완료)
    Story: 나는 기사로서 각 단계를 버튼 하나로 완료 처리하고 싶다

Epic 2: 실시간 위치 추적
  Feature 2-1: 기사 GPS 송신 (3초 간격, 이동 중만)
  Feature 2-2: 고객 앱 기사 위치 수신
  Feature 2-3: 어드민 전체 기사 지도 표시

Epic 3: 어드민
  Feature 3-1: 기사 현황 지도 (클러스터링)
  Feature 3-2: 주문 현황 목록 조회

[미결 기능 — 다음 미팅 확정 필요]
  - 앱 내 결제 (PG 연동) → 1차 스코프 포함 여부 미결
  - 주문 취소 정책 → 세부 규칙 미결
```

## 5. `acceptance-criteria-generator`

**역할**: 각 Feature에 대해 테스트 가능한 수락 기준(AC)을 Given-When-Then 형식으로 생성한다.

**입력**
```
- Feature 목록 (feature-distiller 출력)
- NFR 목록 (nfr-extractor 출력, 있는 경우)
```

**처리 단계**
```
1. Feature별로 주요 시나리오 식별 (Happy Path + 대표 예외)
2. 각 시나리오를 Given-When-Then으로 서술
   - Given: 사전 조건
   - When: 사용자 행동 또는 시스템 이벤트
   - Then: 기대 결과 (측정 가능한 표현 사용)
3. 성능 조건이 있는 경우 Then에 수치 포함
4. "모호한 표현" 감지 후 구체화
   - "빠르게 표시" → "3초 이내 렌더링"
   - "정상적으로 작동" → 구체적 동작 명시 요구
```

**출력 예시**
```
[Feature 2-1: 기사 GPS 송신]

AC-1 (Happy Path)
  Given: 기사가 앱에 로그인하고 이동 중인 상태
  When:  3초가 경과한다
  Then:  현재 GPS 좌표가 서버에 전송된다
         전송 성공 여부가 앱 내 상태 아이콘으로 표시된다

AC-2 (정지 중 스킵)
  Given: 기사가 정지한 상태 (속도 < 2km/h)
  When:  3초가 경과한다
  Then:  GPS 데이터가 전송되지 않는다
         배터리 소모 감소 목적

AC-3 (네트워크 끊김)
  Given: 기기가 오프라인 상태
  When:  위치 전송을 시도한다
  Then:  전송 실패 경보를 표시하지 않는다 (silent fail)
         네트워크 복구 시 자동 재전송한다

[Feature 3-1: 어드민 기사 지도]

AC-1 (지도 로딩)
  Given: 운영자가 어드민에 로그인한 상태
  When:  기사 현황 지도 메뉴를 클릭한다
  Then:  3초 이내에 현재 활성 기사 전원의 위치가 지도에 표시된다

AC-2 (클러스터링)
  Given: 동일 반경 500m 내에 기사가 3명 이상 있다
  When:  지도 줌 레벨이 14 이하다
  Then:  개별 핀 대신 숫자가 표시된 클러스터 핀으로 그룹핑된다
```

## 6. `nfr-extractor`

**역할**: CPS Context의 제약 조건과 수치 기준에서 비기능 요구사항을 추출하고 카테고리별로 정리한다.

**입력**
```
- CPS Context 누적 전체
- 고객사 업종 (규정 기반 NFR 추론용)
```

**처리 단계**
```
1. Context에서 아래 키워드 패턴 스캔:
   - 성능: "빠르게", "지연", "응답", 시간 수치, 트래픽 수치
   - 가용성: "안 터지게", "다운타임", 피크타임 언급
   - 보안: "권한", "로그인", "인증", 규정 언급
   - 확장성: 사용자 증가 계획, 데이터 증가 예측
   - 접근성: 기기 사양, 비기술직 언급, 저사양 환경
2. 각 항목을 카테고리 분류: 성능 / 가용성 / 보안 / 확장성 / 운영성 / 접근성
3. 수치가 명시된 경우 → 그대로 사용
   수치가 암묵적인 경우 → 업종 표준 기준 제안 (출처 명기)
4. 측정 방법 정의
```

**출력 예시**
```
[비기능 요구사항]

성능
  NFR-P1: API 응답시간 p99 < 300ms (피크타임 기준)
           측정: APM 대시보드 (DataDog 또는 동급)
           근거: Meeting 3 — "피크타임에 절대 느리면 안 된다"

  NFR-P2: 기사 위치 갱신 주기 ≤ 3초
           측정: 클라이언트 수신 타임스탬프 로그
           근거: Meeting 3 — "위치 업데이트가 느리다는 불만"

가용성
  NFR-A1: 피크타임(12-13시, 18-20시) 가용성 99.9% 이상
           측정: 월별 다운타임 집계
           근거: Meeting 2 — "그때 절대 안 터져야"

확장성
  NFR-S1: 기사 수 150명까지 무중단 확장 가능
           측정: 부하 테스트 (기사 150명 동시 시뮬레이션)
           근거: Meeting 2 — "6개월 내 150명 목표"

접근성
  NFR-U1: 어드민 주요 기능은 IT 비전문가가 별도 교육 없이 사용 가능
           측정: 운영팀 대상 사용성 테스트 (완료율 80% 이상)
           근거: Meeting 4 — "운영팀이 비기술직"
```

## 7. `scope-boundary-detector`

**역할**: 미팅 전반에서 명시·암묵적으로 스코프에서 제외된 항목을 감지하고 경계를 확정한다.

**입력**
```
- CPS 전체 (Context + Problem + Solution + 보류 항목)
- 미팅 로그 원문 (발화 레벨 접근 가능한 경우)
```

**처리 단계**
```
1. 아래 패턴의 발화 스캔:
   - 명시적 제외: "그건 이번에 안 해도 돼요", "1차엔 빼죠", "나중에"
   - 암묵적 제외: 논의됐지만 결론 없이 다음으로 넘어간 항목
   - 전제 합의: "당연히 X는 되겠죠" 형태로 넘어간 항목
2. 각 항목을 아래 3가지로 분류:
   - In Scope: 확정된 포함 항목
   - Out of Scope: 명시적으로 제외 합의된 항목
   - TBD: 아직 결정되지 않은 항목
3. Out of Scope에 제외 근거(발언 출처) 태깅
4. TBD에 결정 필요 시점 또는 결정 조건 명시
```

**출력 예시**
```
[스코프 경계]

In Scope (1차)
  ✅ 고객 앱 — 주문, 실시간 위치 추적, 주문 이력
  ✅ 기사 앱 — 주문 수신, 상태 업데이트, GPS 송신
  ✅ 어드민 — 기사 현황 지도, 주문 목록 조회
  ✅ 실시간 알림 — WebSocket 기반 (확정)

Out of Scope (1차 제외, 근거 포함)
  ❌ 앱 내 결제 (PG 연동)
     근거: Meeting 1 — "결제는 일단 외부 링크로, 나중에 붙이죠"
  ❌ 리뷰·평점 시스템
     근거: Meeting 2 — "MVP엔 필요 없을 것 같아요"
  ❌ 다국어 지원
     근거: (암묵적) 전 미팅에서 한 번도 언급 없음

TBD (다음 미팅 결정 필요)
  ❓ 주문 취소 정책 — 고객/기사 취소 가능 시점, 패널티 규칙
     결정 필요 시점: 기사 앱 개발 시작 전
  ❓ 어드민 권한 체계 — 관리자 역할 분리 여부
     결정 필요 시점: 어드민 설계 시작 전
```

## 8. `prd-drafter`

**역할**: 앞선 6개 증류 스킬의 출력을 받아 CPS 출처 태그가 붙은 PRD 초안을 생성한다.

**입력**
```
- persona-extractor 출력
- goal-metric-definer 출력
- feature-distiller 출력
- acceptance-criteria-generator 출력
- nfr-extractor 출력
- scope-boundary-detector 출력
- CPS 전체 (원본 참조용)
```

**처리 단계**
```
1. PRD 표준 구조로 섹션 조립
2. 각 항목에 CPS 출처 태그 삽입
   형식: [출처: Meeting N, Context/Problem/Solution]
3. 출처가 없는 항목은 생성하지 않고 TBD로 표기
4. 확정되지 않은 기술 결정은 "결정 필요" 플래그 삽입
5. 전체 문서 일관성 검토 (용어 통일, 중복 제거)
```

**출력 구조**

```markdown
# PRD: [프로젝트명] v0.1 (초안)

## 3. 사용자 페르소나

### Epic 1: ...
  Feature 1-1 [출처: Meeting X, Solution]
    AC: ...

## 5. 비기능 요구사항
[출처: Meeting N, Context]

## 6. 스코프 경계
### In Scope
### Out of Scope [근거 출처 포함]
### TBD

## 7. 의존성 및 제약 조건

## 8. 미결 사항
[결정 필요 항목 + 결정 필요 시점]
```

## 9. `requirement-validator`

**역할**: PRD 초안의 모든 요구사항이 CPS에 출처가 있는지 역방향 검증하고, CPS의 확정 결정이 PRD에 누락 없이 반영되었는지 순방향 검증한다.

**입력**
```
- PRD 초안 (prd-drafter 출력)
- CPS 전체 (비교 기준)
```

**처리 단계**
```
1. 역방향 검증 (PRD → CPS)
   PRD의 각 요구사항에 대해:
   - CPS에 동일하거나 유사한 내용이 있는가?
   - 없으면 → HALLUCINATED 플래그

2. 순방향 검증 (CPS → PRD)
   CPS의 각 확정 결정에 대해:
   - PRD에 반영되었는가?
   - 없으면 → MISSING 플래그

3. 수치 일치 검증
   PRD의 수치(응답시간, 사용자 수 등)가 CPS 수치와 일치하는가?
   - 불일치 시 → MISMATCH 플래그

4. 검증 결과 리포트 생성
```

**출력 예시**
```
[검증 리포트]

역방향 검증 (PRD → CPS)
  ✅ VALID   기사 앱 주문 수신 [Meeting 3, Solution]
  ✅ VALID   WebSocket 알림 [Meeting 3, Solution 확정]
  ❌ HALLUCINATED  "다국어 지원" — CPS에 출처 없음 → 삭제 권고
  ⚠️ MISMATCH  NFR-P1: PRD에 "200ms" 기재 / CPS에는 "300ms" → 확인 필요

순방향 검증 (CPS → PRD)
  ✅ REFLECTED  Redis pub/sub 위치 이벤트 [Meeting 3]
  ❌ MISSING    SSE 어드민 집계 스냅샷 [Meeting 4] → PRD에 미반영
  ❌ MISSING    Kakao Map 클러스터링 [Meeting 4] → PRD에 미반영

수치 일치
  ⚠️ 기사 수: PRD "100명" / CPS "150명" → 불일치

최종: VALID 4 / HALLUCINATED 1 / MISSING 2 / MISMATCH 2
권고: 재검토 후 prd-drafter 재실행
```

## 10. `prd-gap-detector`

**역할**: PRD 초안에서 섹션 누락, TBD 항목, 측정 불가 표현, 결정 없이 넘어간 항목을 감지하고 보완 질문을 생성한다.

**입력**
```
- PRD 초안
- PRD 표준 섹션 체크리스트
```

**처리 단계**
```
1. 필수 섹션 존재 여부 확인
   (배경 / 목표·KPI / 페르소나 / 기능 요구사항 /
    NFR / 스코프 경계 / 의존성 / 미결 사항)

2. 모호한 표현 감지 (정규식·키워드 기반)
   - "빠르게", "쉽게", "편리하게", "충분히", "적절히"
   - "필요한 경우", "추후 결정", "TBD" (미보완 상태)

3. AC 누락 기능 감지
   AC 없이 Feature만 기술된 항목 목록화

4. 보완 질문 자동 생성
   - 모호 표현 → "구체적으로 어떤 기준인가요?"
   - TBD → "언제까지 결정이 필요한가요?"
   - 수치 없는 NFR → "측정 기준은 무엇인가요?"

5. 다음 미팅 안건으로 전환
```

**출력 예시**
```
[갭 감지 리포트]

섹션 누락
  ❌ "의존성 및 제약 조건" 섹션 없음

모호한 표현 (7건)
  ⚠️ Feature 1-1: "빠르게 주문 완료" → "몇 초 이내?"
  ⚠️ NFR-U1: "편리하게 사용" → 사용성 측정 기준 필요
  ⚠️ Feature 3-2: "적절한 권한 체계" → 권한 롤 목록 필요

AC 누락 기능 (3건)
  - Feature 1-3: 주문 상태 관리
  - Feature 2-2: 고객 앱 기사 위치 수신
  - Feature 3-2: 주문 현황 목록 조회

다음 미팅 보완 안건
  1. 주문 완료 목표 응답시간 수치 확정
  2. 어드민 권한 롤 목록 합의
  3. 주문 취소 정책 세부 규칙 결정
```

## 11. `dependency-mapper`

**역할**: 기능 간 선행 관계와 외부 시스템 의존성을 매핑하여 개발 순서 제약과 통합 리스크를 명시한다.

**입력**
```
- Feature 목록 (feature-distiller 출력)
- 확정된 기술 스택
- Out of Scope 목록 (연동 불필요 항목 제외용)
```

**처리 단계**
```
1. 기능 간 선행 관계 분석
   - A 기능이 완성되어야 B 기능 개발 가능한 경우 연결
   - 예: 기사 GPS 송신 → 고객 앱 위치 수신 → 어드민 지도 표시

2. 외부 시스템 연동 의존성 파악
   - 외부 API (지도, 알림, 결제 등)
   - 외부 인증 시스템
   - 레거시 시스템 마이그레이션

3. 의존성 유형 분류
   - 내부 의존: 기능 간 (개발 순서 제약)
   - 외부 의존: 서드파티 서비스 (계약·테스트 환경 제약)
   - 데이터 의존: 선행 데이터 없으면 구현 불가

4. 크리티컬 패스 식별 (지연 시 전체 일정에 영향)
```

**출력 예시**
```
[의존성 맵]

기능 간 의존 (개발 순서 제약)
  [기사 GPS 송신] ──▶ [고객 앱 위치 수신] ──▶ [어드민 기사 지도]
  [주문 생성] ──▶ [기사 주문 수신] ──▶ [주문 상태 업데이트]
  [사용자 인증] ──▶ (모든 기능)

외부 시스템 의존
  Kakao Map API
    - 사용 기능: 어드민 기사 지도, 고객 앱 배달 현황 지도
    - 선결 사항: API 키 발급, 월 한도 산정
    - 리스크: 무료 한도 초과 시 과금 발생

  FCM (Firebase Cloud Messaging) — 알림 보조용
    - 사용 기능: 기사 앱 백그라운드 알림
    - 리스크: iOS 알림 권한 거절 시 대체 방안 필요

크리티컬 패스
  사용자 인증 → 주문 생성 → 기사 주문 수신
  (이 3개 중 하나 지연 시 전체 데모 불가)
```

## 12. `prd-change-detector`

**역할**: PRD 확정 이후 미팅 발화를 실시간 모니터링하여 스코프 변경·요구사항 충돌·신규 요구를 감지하고 영향도를 분석한다.

**입력**
```
- 현재 미팅 발화 (실시간)
- 확정된 PRD
- CPS 누적
```

**처리 단계**
```
1. 발화에서 PRD 관련 키워드 감지
   - 기능명, 스코프 항목, NFR 수치 언급
   - "~도 해줘요", "~은 바꿔야 할 것 같아요" 패턴

2. 감지된 발화를 PRD와 대조
   - 기존 In Scope 항목과 충돌하는가?
   - Out of Scope 항목을 다시 꺼내는가?
   - PRD에 없는 신규 요구인가?
   - NFR 수치를 변경하는가?

3. 변경 유형 분류
   - SCOPE_CHANGE: Out of Scope 항목 재요청
   - REQUIREMENT_CHANGE: 기존 기능 동작 변경
   - NFR_CHANGE: 비기능 요구사항 수치 변경
   - NEW_REQUIREMENT: PRD에 없는 신규 요구

4. 영향도 분석
   - 일정, 비용, 기술 결정에 미치는 영향
   - 선행 기능 재설계 필요 여부

5. 실시간 FDE 경보 생성
```

**출력 예시**
```
⚠️ PRD 변경 감지 [실시간]

유형: SCOPE_CHANGE
발화: "아 참, 앱 내 결제도 이번에 같이 넣을 수 있을까요?"
      (Meeting 5, 14:23)

PRD 현황:
  Out of Scope — "앱 내 결제 (PG 연동)"
  근거: Meeting 1 — "결제는 일단 외부 링크로"

영향도 분석:
  일정: PG사 계약 + 연동 개발 약 3-4주 추가
  기술: 결제 모듈 아키텍처 추가 설계 필요
  규정: 전자금융업 신고 여부 확인 필요 (업종에 따라 다름)
  선행 영향: 주문 생성 Flow 재설계 필요

FDE 권장 액션:
  "앱 내 결제 추가 시 일정과 비용 영향이 큽니다.
   1차 스코프 포함 여부를 오늘 확정해주세요."
```

## 13. `traceability-matrix-builder`

**역할**: PRD의 모든 요구사항과 CPS 출처를 연결하는 추적성 매트릭스를 생성한다. 이후 요구사항 변경 시 영향 범위를 즉시 파악할 수 있게 한다.

**입력**
```
- PRD 확정본
- CPS 전체 (미팅별 로그 포함)
```

**처리 단계**
```
1. PRD의 모든 요구사항 ID 목록 추출
2. 각 요구사항에 대해:
   - 출처 미팅 번호
   - 발화 원문 (또는 요약)
   - CPS 차원 (Context / Problem / Solution)
   - 연결된 다른 요구사항 ID
3. 매트릭스 형태로 조립
4. 변경 영향 체인 계산
   - 요구사항 A가 변경되면 영향받는 요구사항 목록 생성
```

**출력 예시**
```
[요구사항 추적성 매트릭스]

| NFR-U1 | 어드민 사용성 | Meeting 4 | Context | "운영팀이 비기술직" | F3-1, F3-2 |

[변경 영향 체인 예시]
NFR-P2 변경 시 → F2-1, F2-2, F2-3, F3-1 재검토 필요
```

```
  prd-drafter (증류 결과 전체 입력)
        │
```
