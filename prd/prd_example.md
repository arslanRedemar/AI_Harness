## 1. 배경 및 목적

### 1-1. 비즈니스 배경

서울 강남구 중심 배달 스타트업. 현재 카카오채널로 수동 주문을 접수하고 있어 운영 효율이 낮고 피크타임(점심 12-13시, 저녁 18-20시) 대응에 한계가 있다.[^M1-C] 이번 프로젝트는 고객 앱·기사 앱·어드민을 갖춘 전용 플랫폼으로의 디지털 전환 첫 단계다.[^M1-C]

### 1-2. 해결 목표

| 현재 상태 | 목표 상태 |
|-----------|----------|
| 카카오채널 수동 주문 접수 | 앱 기반 주문 처리 전환율 80% 이상 |
| 수동 기사 배정 | 앱에서 주문 수신·수락·배달 완료 처리 |
| 기사 위치 파악 불가 | 실시간 GPS 기반 위치 추적 (3초 갱신) |
| 운영팀 현황 파악 전화·메신저 의존 | 어드민 지도에서 전체 기사 실시간 모니터링 |

## 4. 기능 요구사항

### Epic 1: 주문 관리

#### Feature 1-1: 고객 앱 주문 생성 [^M1-S]

**User Story**
> 나는 고객으로서 앱에서 메뉴를 선택하고 주문을 넣어 배달을 요청하고 싶다.

**수락 기준 (AC)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | 고객이 앱에 로그인한 상태 | 메뉴 선택 후 주문하기를 탭 | 주문이 생성되고 접수 상태로 전환된다 |
| AC-2 | 주문이 생성된 상태 | — | 고객 화면에 주문 번호와 예상 배달 시간이 표시된다 |
| AC-3 | 네트워크 오류 발생 | 주문하기를 탭 | 오류 메시지가 표시되고 주문은 생성되지 않는다 (중복 방지) |

#### Feature 1-2: 기사 앱 주문 수신 및 수락 [^M2-S]

**User Story**
> 나는 기사로서 새 주문이 들어오면 즉시 알림을 받아 수락 또는 거절하고 싶다.

**수락 기준 (AC)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | 기사가 앱에 로그인하고 대기 상태 | 새 주문이 배정된다 | 3초 이내에 알림이 도착하고 수락/거절 버튼이 표시된다 |
| AC-2 | 수락/거절 화면이 표시된 상태 | 수락을 탭 | 주문 상태가 "기사 배정"으로 전환되고 픽업 주소가 표시된다 |
| AC-3 | 수락/거절 화면이 표시된 상태 | 30초 이내에 응답 없음 | 자동으로 다음 기사에게 재배정된다 |

#### Feature 1-3: 주문 상태 관리 [^M2-S]

**User Story**
> 나는 기사로서 각 배달 단계를 버튼 하나로 완료 처리하고 싶다.

**수락 기준 (AC)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | 기사가 픽업 장소에 도착 | "픽업 완료" 버튼을 탭 | 주문 상태가 "배달 중"으로 전환되고 고객에게 알림이 전송된다 |
| AC-2 | 기사가 배달 완료 | "배달 완료" 버튼을 탭 | 주문 상태가 "완료"로 전환된다 |

### Epic 2: 실시간 위치 추적

#### Feature 2-1: 기사 GPS 송신 [^M3-S]

**User Story**
> 나는 기사로서 이동 중에 내 위치가 자동으로 전송되어 고객이 확인할 수 있기를 원한다.

**수락 기준 (AC)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | 기사가 로그인하고 이동 중 | 3초가 경과 | 현재 GPS 좌표가 서버에 전송된다 |
| AC-2 | 기사가 정지 상태 (속도 < 2km/h) | 3초가 경과 | GPS 데이터가 전송되지 않는다 (배터리 절약) |
| AC-3 | 기기가 오프라인 상태 | 위치 전송 시도 | 앱에 오류 표시 없이 silent fail, 복구 시 자동 재전송 |

#### Feature 2-2: 고객 앱 기사 위치 수신 [^M3-S]

**User Story**
> 나는 고객으로서 주문 후 배달 화면에서 기사 위치를 실시간으로 보고 싶다.

**수락 기준 (AC)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | 주문이 "배달 중" 상태 | 고객이 배달 현황 화면 진입 | 지도에 기사 위치 핀이 표시된다 |
| AC-2 | 기사가 이동 중 | — | 핀이 3초 이내 간격으로 갱신된다 |

**수락 기준 (AC)**

| # | Given | When | Then |

**수락 기준 (AC)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-1 | 운영자가 주문 현황 메뉴 진입 | — | 접수·배달중·완료 상태별 주문 목록이 표시된다 |
| AC-2 | — | 주문 상태 변경 발생 | 새로고침 없이 목록이 자동 갱신된다 |

## 7. 의존성 및 제약

### 기능 간 선행 관계

```
[사용자 인증]
    │
    ├──▶ [주문 생성] ──▶ [기사 주문 수신] ──▶ [주문 상태 관리]
    │                          │
    └──▶ [기사 GPS 송신] ──▶ [고객 위치 수신] ──▶ [어드민 기사 지도]
```

### 외부 시스템 의존

| 시스템 | 사용 기능 | 선결 사항 | 리스크 |
|--------|---------|---------|--------|
| Kakao Map API | 기사 지도, 배달 현황 지도 | API 키 발급, 월 한도 산정 | 무료 한도(30만 건/월) 초과 시 과금 |
| FCM | 기사 앱 백그라운드 알림 | Firebase 프로젝트 설정 | iOS 알림 권한 거절 시 대체 방안 필요 |
| WebSocket 서버 | 실시간 위치·알림 전체 | Redis pub/sub 설정 | 연결 수 증가 시 서버 메모리 압박 |

### 크리티컬 패스

> **사용자 인증 → 주문 생성 → 기사 주문 수신**
> 이 3개 중 하나라도 지연되면 전체 데모 불가
